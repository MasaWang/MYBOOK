## **11.1 Purpose**

## **11.1 目的**

**EN  
**This section defines the cloud’s role without making the system
cloud-dependent for all core local behavior.

**ZH-TW  
**本章在不使所有核心本地行為依賴雲端的前提下，定義雲端角色。

## **11.2 Cloud Architecture Definition**

## **11.2 雲端架構定義**

**EN  
**Cloud Architecture defines how OceanAI Cloud participates in the
OceanAI / Artibird system. It provides remote services, account
management, synchronization, lifecycle services, historical data,
alerts, analytics, multi-site management, and commercial service support
where applicable.

OceanAI Cloud extends the system beyond the local environment. It should
enhance supported system behavior but should not be treated as the
required execution path for every core local function.

**ZH-TW  
**雲端架構定義 OceanAI Cloud 如何參與 OceanAI / Artibird
系統。它在適用條件下提供遠端服務、帳號管理、同步、生命週期服務、歷史數據、警報、分析、多站點管理與商業服務支援。

OceanAI Cloud
將系統能力擴展至本地環境之外。它應增強受支援系統行為，但不應被視為每一項核心本地功能的必要執行路徑。

## **11.4 Account Management**

## **11.4 帳號管理**

**EN  
**Account management defines how users, roles, permissions,
organizations, homes, sites, rooms, and devices are associated within
OceanAI Cloud.

Account management may support owner, admin, user, installer, guest, or
organization-level roles where implemented. Account-based access must
follow authentication, authorization, privacy, and security
requirements.

**ZH-TW  
**帳號管理定義用戶、角色、權限、組織、住宅、站點、房間與設備如何在
OceanAI Cloud 中建立關聯。

帳號管理可在已實作條件下支援 Owner、Admin、User、Installer、Guest
或組織級角色。基於帳號的存取必須遵循身份認證、授權、隱私與安全要求。

## **11.5 Device Binding**

## **11.5 設備綁定**

**EN  
**Device binding defines how supported devices are registered and
associated with a user account, home, site, room, organization, or
system deployment.

Device binding may be performed through Artibird App, Artibird Hub,
Artibird Gate, or supported cloud workflows depending on implementation.
Binding behavior must follow device identity, account permission,
security, and validation requirements.

**ZH-TW  
**設備綁定定義受支援設備如何被註冊並關聯至用戶帳號、住宅、站點、房間、組織或系統部署。

依實作方式不同，設備綁定可透過 Artibird App、Artibird Hub、Artibird Gate
或受支援雲端流程完成。綁定行為必須遵循設備身份、帳號權限、安全與驗證要求。

| **Binding Item** | **Description** |

| Account association | Links device to user or organization account |
| Site association | Links device to a home, site, building, classroom, or campus |
| Room assignment | Assigns device to a room, area, or zone |
| Permission check | Verifies user authority to bind or modify device |
| Security verification | Confirms trusted binding behavior where supported |
| Binding record | Stores binding result and lifecycle record where supported |

## **中文表**

| 安全驗證     | 在支援條件下確認可信綁定行為             |
| 綁定記錄     | 在支援條件下儲存綁定結果與生命週期記錄   |

## **11.6 Remote Access**

## **11.6 遠端存取**

**EN  
**Remote access allows users to monitor or control supported devices,
rooms, scenes, automations, or sites outside the local network where
cloud service, account binding, permission rules, and network conditions
are available.

Remote access should be treated as a cloud-enabled capability. It
extends user control beyond the local environment but does not replace
local runtime control where supported local components are present.

**ZH-TW  
**遠端存取允許用戶在雲端服務、帳號綁定、權限規則與網路條件可用時，於本地網路之外監測或控制受支援設備、房間、場景、自動化或站點。

遠端存取應被視為雲端啟用能力。它將用戶控制擴展至本地環境之外，但不取代所需本地元件具備時的本地
Runtime 控制。

| Remote scene control | Execute supported scenes remotely |
| Remote status view | View device state, room state, or site status |
| Remote alert access | Receive alerts or notifications where supported |
| Remote configuration | Modify supported settings or rules where allowed |
| Multi-site access | Access multiple homes, classrooms, buildings, or campuses where supported |

## **中文表**

| **遠端存取功能** | **說明**                                 |

| 遠端狀態查看     | 查看設備狀態、房間狀態或站點狀態         |

## **11.7 Data Synchronization**

## **11.7 數據同步**

**EN  
**Data synchronization defines how selected state, configuration, logs,
lifecycle data, historical records, account data, and alert records may
synchronize between the local runtime system and OceanAI Cloud.

Synchronization may be continuous, scheduled, event-triggered, manual,
or delayed depending on implementation, network availability, privacy
rules, account binding, and cloud service availability.

**ZH-TW  
**數據同步定義特定狀態、配置、日誌、生命週期數據、歷史記錄、帳號數據與警報記錄如何在本地
Runtime 系統與 OceanAI Cloud 之間同步。

依實作方式、網路可用性、隱私規則、帳號綁定與雲端服務可用性不同，同步可採用連續、定時、事件觸發、手動或延遲方式。

## **English Table**

| **Sync Data Type** | **Direction** | **Description** |
|----|----|----|
| Device state | Hub ↔ Cloud | Selected device states may synchronize |
| Configuration | Hub ↔ Cloud | Device, room, scene, and automation configuration may synchronize |
| Lifecycle data | Hub ↔ Cloud | Firmware, software, and device lifecycle data may synchronize |
| Logs | Hub → Cloud | Runtime, diagnostic, security, or failure logs may synchronize where supported |
| Historical records | Hub → Cloud | Historical data may synchronize where enabled |
| Account data | Cloud → Hub / App | Account, role, and permission data may synchronize |
| Alert records | Hub / Cloud → App | Alert records may be delivered or stored where supported |

## **中文表**

| **同步數據類型** | **方向** | **說明** |
|----|----|----|
| 設備狀態 | Hub ↔ Cloud | 特定設備狀態可同步 |
| 配置 | Hub ↔ Cloud | 設備、房間、場景與自動化配置可同步 |
| 生命週期數據 | Hub ↔ Cloud | 韌體、軟體與設備生命週期數據可同步 |
| 日誌 | Hub → Cloud | 支援條件下 Runtime、診斷、安全或故障日誌可同步 |
| 歷史記錄 | Hub → Cloud | 啟用條件下歷史數據可同步 |
| 帳號數據 | Cloud → Hub / App | 帳號、角色與權限數據可同步 |
| 警報記錄 | Hub / Cloud → App | 警報記錄可在支援條件下傳送或儲存 |

## **11.9 歷史數據與分析**

multi-site analytics.

Historical data and analytics must follow data retention, privacy,
account permission, storage, and validation boundaries.

**ZH-TW

## **11.10 警報與通知**

Alert delivery depends on cloud availability, App permission, account
binding, notification settings, event type, and validation status.

警報傳送取決於雲端可用性、App
權限、帳號綁定、通知設定、事件類型與驗證狀態。

## **11.11 Multi-Site Management**

homes, classrooms, offices, buildings, campuses, infrastructure sites,
or other supported deployment environments under one account or
organization structure.

OceanAI Cloud may provide cross-site visibility, account management,
device grouping, remote monitoring, alert management, reporting,
configuration synchronization, and lifecycle management where supported.

**ZH-TW  
**多站點管理允許用戶或組織在同一帳號或組織結構下，管理多個住宅、教室、辦公室、建築、校園、基礎設施站點或其他受支援部署環境。

OceanAI Cloud
可在支援條件下提供跨站點可視化、帳號管理、設備分組、遠端監測、警報管理、報告、配置同步與生命週期管理。

## **11.12 Commercial Service Plans**

## **11.12 商業服務方案**

**EN  
**Commercial service plans may be enabled by OceanAI Cloud where
applicable. These plans may support enhanced remote services, historical
analytics, multi-site management, reporting, enterprise support,
lifecycle management, or other cloud-enhanced services.

Commercial service plans must remain aligned with actual cloud
capabilities, implemented service levels, privacy policy, data retention
rules, and validated system behavior.

**ZH-TW  
**在適用條件下，商業服務方案可由 OceanAI Cloud
啟用。這些方案可支援增強型遠端服務、歷史分析、多站點管理、報告、企業支援、生命週期管理或其他雲端增強服務。

商業服務方案必須與實際雲端能力、已實作服務等級、隱私政策、數據保留規則與已驗證系統行為保持一致。

## **11.13 Cloud Failure Behavior**

## **11.13 雲端故障行為**

**EN  
**Cloud failure behavior defines how the system behaves when OceanAI
Cloud is unavailable, degraded, delayed, or unreachable.

When cloud services are unavailable, supported local functions should
continue where required local components are present. Remote access,
cloud synchronization, cloud alerts, multi-site management, cloud
analytics, account updates, and cloud lifecycle services may be
unavailable, delayed, or degraded.

**ZH-TW  
**雲端故障行為定義 OceanAI Cloud
不可用、降級、延遲或無法連接時，系統如何運作。

當雲端服務不可用時，在所需本地元件具備條件下，受支援本地功能應繼續運作。遠端存取、雲端同步、雲端警報、多站點管理、雲端分析、帳號更新與雲端生命週期服務可能不可用、延遲或降級。

## **English Table**

| **Cloud Condition** | **Expected Behavior** |
|----|----|
| Cloud available | Remote access, synchronization, alerts, analytics, and lifecycle services operate where supported |
| Cloud delayed | Synchronization, alerts, and remote status may be delayed |
| Cloud unavailable | Supported local functions continue where local components are present |
| Internet unavailable | Remote access and cloud synchronization may stop; local runtime may continue |
| Account service unavailable | Account updates, binding changes, or role changes may be delayed |
| Cloud sync failure | Local data may queue, retry, or remain local depending on implementation |
| Cloud alert failure | Remote notifications may be delayed or unavailable |

## **中文表**

| **雲端狀態** | **預期行為** |
|----|----|
| 雲端可用 | 在支援條件下，遠端存取、同步、警報、分析與生命週期服務運作 |
| 雲端延遲 | 同步、警報與遠端狀態可能延遲 |
| 雲端不可用 | 本地元件具備時，受支援本地功能繼續 |
| Internet 不可用 | 遠端存取與雲端同步可能停止；本地 Runtime 可繼續 |
| 帳號服務不可用 | 帳號更新、綁定變更或角色變更可能延遲 |
| 雲端同步失敗 | 本地數據可依實作方式排隊、重試或保留本地 |
| 雲端警報失敗 | 遠端通知可能延遲或不可用 |

## **11.14 Cloud Security and Privacy Boundary**

## **11.14 雲端安全與隱私邊界**

**EN  
**Cloud services must follow defined security and privacy requirements.
These include account authentication, role-based access control, secure
communication, data retention, user permission, device binding security,
audit logging where supported, and privacy policy compliance.

Cloud participation must not be used to imply unlimited data collection,
unrestricted remote access, unsupported AI processing, or undefined
sharing of user data.

**ZH-TW  
**雲端服務必須遵循已定義安全與隱私要求。這包括帳號身份認證、角色權限控制、安全通訊、數據保留、用戶權限、設備綁定安全、支援條件下的審計日誌，以及隱私政策合規。

不得使用雲端參與來暗示無限制數據收集、不受限制遠端存取、未支援 AI
處理或未定義的用戶數據共享。

## **11.15 Cloud Boundary**

## **11.15 雲端邊界**

**EN  
**Cloud architecture must be described according to implemented cloud
services, supported products, software versions, account configuration,
network conditions, privacy policy, security requirements, and
validation evidence.

Cloud architecture does not imply that all functions require cloud, that
all functions operate without local components, that remote access is
always available, or that all third-party services are supported.

**ZH-TW  
**雲端架構必須依已實作雲端服務、受支援產品、軟體版本、帳號配置、網路條件、隱私政策、安全要求與驗證證據進行描述。

雲端架構不表示所有功能都需要雲端、不表示所有功能可以脫離本地元件運作、不表示遠端存取永遠可用，也不表示支援所有第三方服務。

## **11.16 Validation Requirements**

## **11.16 驗證要求**

**EN  
**Any cloud claim involving remote access, synchronization, account
binding, alert delivery, historical analytics, multi-site management,
firmware lifecycle, security, privacy, reliability, or commercial
service plans must be validated under defined conditions.

Validation should include account status, user role, cloud availability,
Internet condition, Hub availability, device state, software version,
firmware version, expected behavior, failure behavior, and recovery
