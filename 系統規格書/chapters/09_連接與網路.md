## **9.2 Connectivity and Networking Definition**

## **9.2 連接與網路定義**

**EN  
**Connectivity and networking define how supported OceanAI / Artibird
components communicate within the local environment and with cloud
services where supported.

The system may include Ethernet, Wi-Fi, BLE Mesh, Wi-Fi Mesh where
implemented and validated, local MQTT communication, App local
connection, cloud connection, gateway communication, and offline local
operation. Connectivity behavior must be defined according to supported
product variants, network conditions, firmware versions, deployment
models, and validation evidence.

**ZH-TW  
**連接與網路定義受支援 OceanAI / Artibird
組件如何在本地環境中通訊，以及在支援條件下如何與雲端服務通訊。

系統可包含 Ethernet、Wi-Fi、BLE Mesh、在已實作並驗證條件下的 Wi-Fi
Mesh、本地 MQTT 通訊、App
本地連接、雲端連接、網關通訊與離線本地運作。連接行為必須依據受支援產品版本、網路條件、韌體版本、部署模型與驗證證據進行定義。

## **9.4 Network Architecture Role**

## **9.4 網路架構角色**

**EN  
**The network architecture provides the communication foundation for
supported runtime behavior. It allows device events, user commands,
sensor data, visual events, execution responses, cloud commands, and
synchronization data to move between supported components.

Artibird Hub provides the local runtime service environment. Artibird
Gate provides supported device access and gateway communication.
Artibird App provides local and remote user access where supported.
OceanAI Cloud provides remote service connectivity. OceanAI Edge
participates through the supported local network path where deployed.

**ZH-TW  
**網路架構為受支援 Runtime
行為提供通訊基礎。它使設備事件、用戶指令、感測數據、視覺事件、執行回應、雲端指令與同步數據能夠在受支援組件之間流動。

Artibird Hub 提供本地 Runtime 服務環境。Artibird Gate
提供受支援設備接入與網關通訊。Artibird App
在支援條件下提供本地與遠端用戶存取。OceanAI Cloud
提供遠端服務連接。OceanAI Edge
在部署條件下透過受支援本地網路路徑參與系統。

## **9.5 Hub Connectivity**

## **9.5 Hub 連接**

**EN  
**Artibird Hub may use Ethernet or Wi-Fi depending on product
configuration and deployment model. Hub connectivity supports local
runtime services, MQTT services, web services, state management,
automation, synchronization, and communication with supported devices,
Gate, App, Cloud, and Edge.

Where reliability is required, Ethernet should be preferred when
available. Wi-Fi connectivity should be validated under defined network
conditions.

**ZH-TW  
**Artibird Hub 可依產品配置與部署模型使用 Ethernet 或 Wi-Fi。Hub
連接支援本地 Runtime 服務、MQTT 服務、Web
服務、狀態管理、自動化、同步，以及與受支援設備、Gate、App、Cloud 與 Edge
的通訊。

在需要較高可靠性的場景中，如條件允許，應優先使用 Ethernet。Wi-Fi
連接應在明確網路條件下完成驗證。

## **9.6 Gate Connectivity**

## **9.6 Gate 連接**

**EN  
**Artibird Gate provides device access and gateway communication for
supported Artibird devices. Depending on product variant and
implementation, Gate may support Wi-Fi, Ethernet, BLE Mesh, and Wi-Fi
Mesh where implemented and validated.

Artibird Gate is a system-level device access gateway. It should not be
described as a simple Wi-Fi accessory. Gate connectivity must be defined
according to supported protocols, device categories, firmware version,
network topology, and capacity validation.

**ZH-TW  
**Artibird Gate 為受支援 Artibird
設備提供設備接入與網關通訊。依產品版本與實作情況不同，Gate
可在已實作並驗證條件下支援 Wi-Fi、Ethernet、BLE Mesh 與 Wi-Fi Mesh。

Artibird Gate 是系統級設備接入網關，不應被描述為簡單 Wi-Fi 配件。Gate
連接能力必須依受支援協議、設備類別、韌體版本、網路拓撲與容量驗證進行定義。

Cloud connection depends on Internet availability, account binding,

## **9.11 Offline Mode**

## **9.11 離線模式**

**EN  
**Offline mode defines which supported local functions can continue when
Internet or cloud access is unavailable. Local device control, supported
local scenes, supported local automation, device state handling, and
physical controls may continue where required local components are
present.

Remote control, cloud synchronization, cloud alerts, account updates,
historical cloud data, and multi-site management may be unavailable or
degraded during offline mode.

**ZH-TW  
**離線模式定義 Internet
或雲端存取不可用時，哪些受支援本地功能可以繼續。本地設備控制、受支援本地場景、受支援本地自動化、設備狀態處理與實體控制，可在所需本地元件具備時繼續運作。

在離線模式下，遠端控制、雲端同步、雲端警報、帳號更新、雲端歷史數據與多站點管理可能不可用或降級。

## **English Table**

| **Function** | **Offline Availability** |
|----|----|

| Local scene execution | Available where scenes are locally configured |
| Local automation | Available where automation is locally supported |
| Physical control | Available depending on device and power condition |
| Device state handling | Available locally where supported |
| Remote control | Not available without cloud / Internet |
| Cloud synchronization | Not available or delayed |
| Multi-site management | Not available or degraded |
| Cloud alerts | Not available or delayed |
| Historical cloud data | Not available or limited |

## **中文表**

## **9.12 Capacity and Connection Limits**

## **9.12 容量與連接限制**

**EN  
**Connectivity capacity must be defined according to validated test
conditions. Capacity-related statements may include max concurrent
device connections, max devices per Gate, max devices per Hub, message
throughput, event latency, local network load, cloud synchronization
load, and reconnection behavior.

Theoretical chip capability must not be used as a final system claim
unless validated under defined hardware, firmware, network, and
deployment conditions.

**ZH-TW  
**連接容量必須依已驗證測試條件定義。容量相關項目可包括最大併發設備連接數、每個

最大設備數、消息吞吐量、事件延遲、本地網路負載、雲端同步負載與重新連接行為。

除非已在明確硬體、韌體、網路與部署條件下完成驗證，否則不得將理論晶片能力作為最終系統宣稱。

## **9.14 Connectivity Boundary**

## **9.14 連接邊界**

**EN  
**Connectivity and networking behavior must be described according to
supported product type, firmware version, hardware version, network
topology, local network quality, cloud availability, gateway
availability, Hub availability, deployment model, and validation
evidence.

Defined connectivity does not imply support for all routers, all
networks, all third-party devices, all protocols, all cloud services, or
all deployment conditions.

**ZH-TW  
**連接與網路行為必須依受支援產品類型、韌體版本、硬體版本、網路拓撲、本地網路品質、雲端可用性、網關可用性、Hub
可用性、部署模型與驗證證據進行描述。

已定義的連接能力不表示支援所有路由器、所有網路、所有第三方設備、所有協議、所有雲端服務或所有部署條件。

## **9.15 Validation Requirements**

## **9.15 驗證要求**

**EN  
**Any networking claim involving capacity, range, latency, reliability,
offline behavior, mesh performance, gateway behavior, cloud
synchronization, MQTT throughput, or concurrent connections must be
validated under defined test conditions.

Validation should include hardware version, firmware version, network
topology, device count, message rate, test duration, failure scenarios,
and environmental conditions where applicable.

**ZH-TW  
**任何涉及容量、範圍、延遲、可靠性、離線行為、Mesh
性能、網關行為、雲端同步、MQTT
吞吐量或併發連接的網路宣稱，都必須在明確測試條件下完成驗證。

驗證應包含硬體版本、韌體版本、網路拓撲、設備數量、消息速率、測試時長、故障場景，以及適用時的環境條件。
