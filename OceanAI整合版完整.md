# **Preface｜前言（Prose Edition｜工程敘事版）**

---

Modern environments are composed of a wide range of devices, including lighting systems, switches, sensors, appliances, and gateways. While these devices have become increasingly intelligent on an individual level, they largely operate as isolated units. Their primary design paradigm remains centered on direct control rather than system-level coordination. As a result, although each device can respond to specific commands, they lack the ability to understand or interact with one another in a meaningful way. This fragmentation prevents environments from functioning as integrated systems and limits the overall intelligence that can be achieved within a space.

現代空間由多種類型的設備構成，包括照明系統、開關、感測器、家電與網關。儘管這些設備在單體層面上日益智能化，但其運作方式仍以彼此獨立為主，設計核心仍然圍繞在「直接控制」而非「系統級協同」。因此，雖然每個設備都能對特定指令做出回應，卻缺乏彼此理解與協作的能力。這種碎片化狀態，使空間無法作為一個整體系統運作，並限制了整體智慧的發展。


OceanAI introduces a fundamentally different perspective. Instead of focusing on how individual devices connect, it addresses how environments operate as cohesive systems. This shift moves the emphasis away from device-level functionality and toward system-level behavior. By redefining the problem at the level of the environment, OceanAI establishes a new foundation for how intelligence can be structured, distributed, and applied across physical spaces.

OceanAI 提出一種根本不同的視角。它不再關注單一設備如何連接，而是關注空間如何作為一個整體系統運作。這一轉變，將焦點從設備功能層轉移到系統行為層。透過在「空間」層級重新定義問題，OceanAI 為智慧的組織、分布與應用建立了一個全新的基礎。


OceanAI is the system that orchestrates intelligence across the physical world. It is not a standalone product, but a system-level intelligence architecture that integrates cloud-based computation, edge intelligence, system orchestration, and data-driven adaptation into a unified operational framework. Within this framework, intelligence is not confined to a single location but is distributed across multiple layers, enabling real-time coordination, contextual awareness, and continuous system evolution.

OceanAI 是統御物理世界的智慧系統。它並非單一產品，而是一個系統級智慧架構，整合雲端運算、邊緣智慧、系統協同與數據驅動的自適應能力，形成一個統一的運作框架。在此框架中，智慧不再集中於單一位置，而是分布於多個層級之間，實現即時協同、語境感知與持續演化。


Artibird represents the physical product layer of the OceanAI system. It provides the tangible interface through which system intelligence is expressed and executed in real-world environments. This layer includes devices such as switches, panels, sensors, gateways, and execution units. Rather than functioning as independent products, these devices operate as coordinated nodes within the broader system, each contributing to perception, control, and system-level response.

Artibird 是 OceanAI 系統的實體產品層，負責將系統智慧轉化為現實世界中的具體表現與執行。該層包含開關、面板、感測器、網關與執行裝置等設備。這些設備不再作為獨立產品運作，而是作為系統中的協同節點，分別承擔感知、控制與系統回應的角色。


The relationship between OceanAI and Artibird defines a clear separation of roles within the architecture. OceanAI establishes the intelligence model and operational logic of the system, while Artibird provides the physical medium through which this intelligence is realized. In this sense, OceanAI defines intelligence, and Artibird expresses intelligence. Together, they form a unified system in which abstract intelligence and physical execution are seamlessly integrated.
 
OceanAI 與 Artibird 之間的關係，構成了架構中的明確分工。OceanAI 負責建立系統的智慧模型與運作邏輯，而 Artibird 則提供實體載體，使這些智慧得以在現實世界中實現。換言之，OceanAI 定義智慧，Artibird 表達智慧。兩者共同形成一個將抽象智慧與物理執行無縫整合的系統。


Through this architecture, OceanAI enables a transition from isolated device control to coordinated environmental operation. Instead of interacting with individual devices, users engage with a system that responds at the level of the environment. This transformation replaces predefined control logic with adaptive intelligence, allowing the system to respond dynamically to context, user behavior, and environmental conditions.
 
透過這一架構，OceanAI 實現了從設備控制到空間協同運作的轉變。使用者不再與單一設備互動，而是與一個能在空間層級做出回應的系統互動。這一轉變以自適應智慧取代預設控制邏輯，使系統能根據語境、使用者行為與環境條件動態調整其運作。


The result of this transformation is the emergence of environments that function as integrated systems rather than collections of devices. In such environments, coordination replaces control, contextual awareness replaces command-based interaction, and intelligence replaces automation. Spaces evolve from passive containers into active systems capable of sensing, responding, and adapting over time.
 
這一轉變的結果，是空間從設備集合演化為整合系統。在這樣的環境中，協同取代控制，語境感知取代指令操作，智慧取代自動化。空間不再是被動的容器，而是具備感知、回應與持續適應能力的主動系統。


OceanAI defines how spaces operate as a system.

## **3.1–3.5 Integrated System Model Description｜整合模型敘事**

**EN**  
The OceanAI system is structured around a three-layer model consisting of Cloud, Edge, and Device layers. This model defines how computation, control, and intelligence are distributed across the system, forming a coherent operational framework for managing heterogeneous environments.

The three-layer system model is not a physical separation of components, but a logical abstraction that determines responsibility boundaries, execution locations, and data flow patterns. It enables the system to balance latency, reliability, and scalability by assigning tasks to the most appropriate layer.

The Cloud Layer represents the global intelligence domain of the system. It is responsible for large-scale data aggregation, long-term storage, cross-environment analytics, and model-level intelligence. The cloud maintains a global view of system states and usage patterns, enabling optimization across multiple spaces and deployments. It also provides remote access capabilities, allowing users and administrators to monitor and control the system beyond the local network.

However, the Cloud Layer does not participate in time-critical control paths. Its role is to extend system capabilities rather than to sustain core operation. By design, OceanAI minimizes dependency on cloud connectivity for real-time functionality.

The Edge Layer functions as the local intelligence domain. It is responsible for real-time inference, contextual data processing, and immediate decision-making. Located close to the physical environment, the edge layer processes sensor inputs, evaluates conditions, and generates control actions with minimal latency.

This layer also supports the deployment of lightweight AI models, such as small language models (SLM) and vision-language models (VLM), enabling localized intelligence without relying on remote computation. By operating at the edge, the system achieves faster response times and improved robustness under unstable network conditions.

The Device Layer consists of distributed physical nodes that interact directly with the environment. These include sensing devices, control interfaces, and actuation units. Devices are responsible for collecting environmental data, executing control commands, and providing interaction endpoints for users.

Within the system model, devices are abstracted as functional entities with standardized interfaces. This abstraction ensures that heterogeneous hardware can be integrated into a unified control and data framework, regardless of protocol or manufacturer.

A key principle of the OceanAI system model is local-first control logic. Control decisions are executed as close to the device layer as possible, typically within the Edge or Artibird Gate environment. This ensures deterministic behavior, low latency, and continued operation even when cloud connectivity is unavailable.

Local-first control logic defines the primary execution path of the system. Events generated by devices are processed locally, evaluated against rules or models, and translated into actions without requiring cloud intervention. This approach significantly reduces system dependency on external infrastructure.

At the same time, OceanAI incorporates a cloud-assisted intelligence model. While real-time decisions are made locally, the cloud contributes to long-term optimization, model training, and global coordination. Data collected from edge environments is aggregated and analyzed in the cloud, where higher-level insights are generated.

These insights are then propagated back to the edge layer in the form of updated models, rules, or configuration parameters. This creates a feedback loop between local execution and global intelligence, enabling continuous system improvement without compromising real-time performance.

The combination of local-first control and cloud-assisted intelligence allows OceanAI to achieve both autonomy and scalability. The system remains operational and responsive at the local level while continuously benefiting from global data and learning.

In summary, the three-layer system model defines how OceanAI distributes intelligence across cloud, edge, and device domains. By clearly separating responsibilities and optimizing execution paths, it establishes a robust and scalable foundation for intelligent environments.

邊緣層同時支援輕量級 AI 模型的部署，例如小型語言模型（SLM）與視覺語言模型（VLM），使系統在不依賴雲端的情況下仍具備智慧能力。透過邊緣運算，系統在網路不穩定環境下依然保持高可靠性。

裝置層由分散的實體節點構成，直接與環境互動，包括感測設備、控制介面與執行裝置。這些設備負責採集環境數據、執行控制指令並提供使用者交互入口。

在系統模型中，所有設備被抽象為具備標準化介面的功能節點，使不同協議與不同廠商的設備能夠整合進統一的控制與數據框架。

OceanAI 的核心原則之一是本地優先控制邏輯。控制決策盡可能在接近裝置層的位置執行，通常位於邊緣或本地中樞環境。這種設計確保系統具備確定性行為、低延遲以及在無雲端連接情況下的持續運作能力。

本地優先控制邏輯構成系統的主要執行路徑。設備產生的事件在本地被處理，經由規則或模型判斷後直接轉化為控制行為，而無需依賴雲端介入，從而顯著降低對外部基礎設施的依賴。

同時，OceanAI 引入雲端輔助智慧模型。在即時決策由本地完成的同時，雲端負責長期優化、模型訓練與全域協同。來自邊緣環境的數據被聚合至雲端進行分析，產生高階洞察。

這些洞察再以模型更新、規則優化或配置參數的形式回傳至邊緣層，形成一個由本地執行與全域智慧組成的回饋循環，使系統在不影響即時性的前提下持續演進。

本地優先控制與雲端輔助智慧的結合，使 OceanAI 同時具備自主性與可擴展性。系統在本地保持穩定運作的同時，持續從全域數據與學習中獲益。

總結而言，三層系統模型定義了 OceanAI 如何在雲端、邊緣與裝置之間分配智慧與控制能力。透過明確的責任劃分與優化的執行路徑，該模型為智慧環境提供了一個穩定且可擴展的基礎。

## **5.1–5.7 Integrated Coordination Architecture（with System–Product Mapping）**

## **8.1–8.7 Integrated Control Architecture Description｜整體控制架構敘事**

**EN**  
The OceanAI control architecture defines how user inputs, system logic, and external interfaces generate and execute control actions within the system. It establishes a unified framework for handling multiple control sources, ensuring consistent behavior, priority resolution, and secure operation.

Control in OceanAI is not limited to a single interface. Instead, it is a multi-source system in which commands may originate from mobile applications, local interfaces, automation logic, voice systems, or centralized management platforms. All control inputs are normalized into a common command model and processed through the coordination layer.

App control represents the primary user interface for system interaction. The Artibird App enables users to monitor device states, issue commands, and configure automation. It supports both local and remote operation. When connected to the local network, the App communicates directly with the Artibird Gate through local APIs or MQTT bridges. When remote, communication is routed through cloud services.

Local Web Server control provides an alternative control path independent of mobile applications. The Artibird Gate hosts a local web interface that allows devices to be accessed and managed through a browser within the same network. This ensures accessibility in environments where mobile connectivity is limited or where direct local control is required.

Voice control integrates natural language interfaces into the system. Commands issued through voice assistants are translated into structured control instructions and injected into the system through the same coordination framework. Voice control operates as an input layer rather than a separate control system, ensuring consistency with other control sources.

Automation control is driven by the intelligence layer. Rules, scenes, and strategies generate control commands based on events, conditions, or learned patterns. These commands are treated equivalently to user-generated commands and are processed through the same coordination and execution pathways.

Manual override is a critical feature of the control architecture. It ensures that direct user input always takes precedence over automated behavior when necessary. For example, physical switches or direct App commands can override active automation rules. This design guarantees user authority and prevents unintended system behavior.

In large-scale deployments such as schools or campuses, centralized control is required. The system supports centralized management interfaces that allow administrators to monitor and control multiple spaces simultaneously. These interfaces operate on top of the same system model, aggregating data and issuing commands across distributed environments.

Multi-role permission control defines access boundaries within the system. Different users are assigned roles with specific permissions, such as administrator, operator, or guest. These roles determine which devices, functions, and control actions are accessible. Permission enforcement is applied at the coordination layer, ensuring that all control requests are validated before execution.

All control actions follow a unified execution path. Regardless of origin, commands are transmitted to the coordination core, evaluated against system rules and permissions, and then dispatched to target devices. This ensures that the system maintains consistent behavior and avoids conflicting control inputs.

The architecture also supports control prioritization and conflict resolution. When multiple control sources issue competing commands, the system resolves conflicts based on predefined priority rules, such as manual override precedence or safety constraints.

In summary, the OceanAI control architecture unifies diverse control interfaces into a single system model. By standardizing control inputs, enforcing permissions, and managing priorities, it ensures reliable, secure, and predictable system operation across all interaction modes.

## **11.1–11.7 Integrated Security and Governance Architecture｜整體安全與治理敘事**

**EN**  
The OceanAI security and governance framework defines how the system ensures data protection, device trust, secure communication, operational safety, and long-term maintainability. It establishes a comprehensive model that integrates security mechanisms with system-level governance, enabling the system to operate reliably in real-world environments.

Security in OceanAI is designed as a multi-layered approach, aligned with the system architecture. It spans local data handling, device identity, communication protocols, access control, firmware lifecycle, and system resilience. These components collectively ensure that the system is not only secure but also governable.

Local data protection is a fundamental principle of the system. Sensitive data, including device states, user interactions, and environmental information, is primarily stored and processed within the local environment. This local-first data model minimizes exposure to external networks and reduces the risk of data leakage. Data transmitted to the cloud is selectively filtered and anonymized when necessary.

Device authentication ensures that only trusted devices can participate in the system. Each Artibird device is assigned a unique identity and must undergo a secure onboarding process before being integrated. Authentication mechanisms may include cryptographic keys, certificates, or secure pairing procedures, preventing unauthorized devices from accessing the system.

Role-Based Access Control (RBAC) defines user permissions within the system. Users are assigned roles such as administrator, operator, or guest, each with specific access rights. These permissions determine which devices, functions, and control actions are allowed. All control requests are validated against the permission model before execution, ensuring controlled and predictable system operation.

Secure MQTT communication is implemented to protect data in transit. Communication between devices, Artibird Gate, and other system components is encrypted using standard protocols such as TLS. Authentication and authorization mechanisms are applied at the broker level, ensuring that only authorized clients can publish or subscribe to specific topics.

Firmware update strategy is critical for maintaining system security and functionality over time. OceanAI supports secure over-the-air (OTA) updates, allowing devices to receive firmware upgrades without manual intervention. Updates are verified through cryptographic signatures to ensure authenticity and integrity before installation.

Fail-safe design ensures that the system remains safe under failure conditions. In the event of hardware malfunction, communication failure, or unexpected system behavior, devices default to predefined safe states. This is particularly important for power control and safety-critical operations, where incorrect behavior could result in damage or risk.

The concept of a governable AIoT framework extends beyond traditional security. It defines how the system can be monitored, controlled, and audited over time. Governance includes logging, state tracking, configuration management, and policy enforcement. It ensures that system behavior remains transparent, traceable, and adjustable.

In the context of AI integration, governance also addresses model behavior and decision accountability. AI-driven decisions are constrained by system rules and policies, ensuring that automated actions remain within defined operational boundaries.

OceanAI integrates security and governance into a unified framework rather than treating them as separate concerns. This approach ensures that the system is not only protected against threats but also maintainable, auditable, and adaptable over its lifecycle.

In summary, the OceanAI security and governance architecture establishes trust at every level of the system. By combining data protection, device authentication, secure communication, controlled access, safe operation, and governance mechanisms, it enables reliable and accountable operation in complex environments.

OceanAI transforms these fragmented components into a coordinated system, where sensing, decision-making, and execution are integrated. While connectivity is necessary, it is not sufficient to achieve system-level behavior. OceanAI builds on connectivity by introducing coordination mechanisms, including event-driven processing, state synchronization, and orchestration logic. These mechanisms allow devices to function collectively under a unified system model.

OceanAI extends this model by incorporating adaptive intelligence through rule engines, inference models, and learning mechanisms. The system can respond not only to explicit inputs but also to contextual conditions and evolving patterns. This enables dynamic and context-aware behavior.

OceanAI introduces a governance framework in which control actions are evaluated within a broader system context. This includes permission management, priority resolution, and policy enforcement. As a result, system behavior becomes consistent, predictable, and auditable.

OceanAI itself functions as the system layer that unifies all components. It sits above devices and below applications, providing the coordination, intelligence, and control logic required for system operation. By abstracting device-specific details into a unified model, OceanAI enables scalability and interoperability across different environments.

In summary, the value of OceanAI is defined by its ability to transform fragmented, device-centric systems into coordinated, intelligent environments. By establishing a unified system layer, it enables consistent operation, scalable deployment, and adaptive behavior across diverse scenarios.
