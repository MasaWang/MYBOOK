# **14. Reliability and Failure Behavior**

# **14. 可靠性與故障行為**

## **14.1 Purpose**

## **14.1 目的**

**EN  
**This section defines how the system behaves under failure, degraded,
offline, and recovery conditions.

**ZH-TW  
**本章定義系統在故障、降級、離線與恢復條件下如何運作。

## **14.2 Reliability and Failure Behavior Definition**

## **14.2 可靠性與故障行為定義**

**EN  
**Reliability and Failure Behavior defines how the OceanAI / Artibird
system maintains supported operation when cloud services, Internet
connection, local runtime services, gateway access, devices, App access,
Edge services, or network conditions become unavailable, degraded,
delayed, or interrupted.

The system should maintain supported local behavior where required local
components are present. When a component is unavailable, the system
should enter a defined degraded state, report status where supported,
and recover according to the expected behavior defined by product type,
deployment condition, and validation result.

**ZH-TW  
**可靠性與故障行為定義當雲端服務、Internet 連接、本地 Runtime
服務、網關接入、設備、App 存取、Edge
服務或網路條件不可用、降級、延遲或中斷時，OceanAI / Artibird
系統如何維持受支援運作。

當所需本地元件具備時，系統應維持受支援本地行為。當某一組件不可用時，系統應進入已定義降級狀態，在支援條件下回報狀態，並依產品類型、部署條件與驗證結果所定義的預期行為恢復。

## **14.3 Failure Behavior Table**

## **14.3 故障行為表**

## **English Table**

| **Failure Scenario** | **Expected Behavior** |
|----|----|
| Cloud unavailable | Local supported functions continue where local components are present |
| Internet unavailable | Local network functions continue where configured |
| Hub unavailable | Device-level behavior may degrade; local runtime functions may stop |
| Gate unavailable | Connected device access may degrade |
| Device offline | Device marked offline; commands queued or rejected depending on design |
| Edge unavailable | Edge-assisted functions stop; basic local runtime continues |
| App unavailable | Physical controls and local automation continue where configured |
| Power failure | Recovery depends on device and Hub startup behavior |
| Network congestion | System may degrade; capacity limits apply |

## **中文表**

|-----------------|-----------------------------------------------|
| 雲端不可用      | 所需本地元件具備時，受支援本地功能繼續運作    |
| Internet 不可用 | 已配置條件下，本地網路功能繼續運作            |
| Hub 不可用      | 設備級行為可能降級；本地 Runtime 功能可能停止 |
| Gate 不可用     | 已接入設備連接可能降級                        |
| 設備離線        | 設備標記為離線；指令依設計排隊或拒絕          |
| Edge 不可用     | Edge 輔助功能停止；基本本地 Runtime 繼續      |
| App 不可用      | 配置條件下，實體控制與本地自動化繼續          |
| 電力故障        | 恢復取決於設備與 Hub 啟動行為                 |
| 網路擁塞        | 系統可能降級；容量限制適用                    |

## **14.4 Cloud Unavailable**

## **14.4 雲端不可用**

**EN  
**Cloud unavailable means OceanAI Cloud cannot be reached, is degraded,
or cannot provide expected cloud services. In this condition, supported
local functions should continue where required local components are
present.

Remote access, cloud synchronization, cloud alerts, account updates,
historical cloud data, multi-site management, cloud analytics, and cloud
lifecycle services may be unavailable, delayed, or degraded.

**ZH-TW  
**雲端不可用指 OceanAI Cloud
無法連接、處於降級狀態，或無法提供預期雲端服務。在此條件下，當所需本地元件具備時，受支援本地功能應繼續運作。

遠端存取、雲端同步、雲端警報、帳號更新、雲端歷史數據、多站點管理、雲端分析與雲端生命週期服務可能不可用、延遲或降級。

## **English Table**

| **Cloud-Dependent Function** | **Behavior When Cloud Is Unavailable** |
|----|----|
| Remote control | Unavailable or degraded |
| Cloud synchronization | Delayed, queued, retried, or stopped depending on implementation |
| Cloud alerts | Delayed or unavailable |
| Account update | Unavailable or delayed |
| Multi-site management | Unavailable or degraded |
| Historical cloud data | Unavailable or limited |
| Local device control | Continues where local components are present |
| Local scene execution | Continues where locally configured and supported |
| Local automation | Continues where locally configured and supported |

## **中文表**

| **雲端依賴功能** | **雲端不可用時行為**             |
|------------------|----------------------------------|
| 遠端控制         | 不可用或降級                     |
| 雲端同步         | 依實作方式延遲、排隊、重試或停止 |
| 雲端警報         | 延遲或不可用                     |
| 帳號更新         | 不可用或延遲                     |
| 多站點管理       | 不可用或降級                     |
| 雲端歷史數據     | 不可用或受限                     |
| 本地設備控制     | 本地元件具備時繼續               |
| 本地場景執行     | 本地已配置且支援時繼續           |
| 本地自動化       | 本地已配置且支援時繼續           |

## **14.5 Internet Unavailable**

## **14.5 Internet 不可用**

**EN  
**Internet unavailable means the local environment has no external
network access. In this condition, local network functions may continue
where the local network, Hub, Gate, ArtiOS, and required devices are
available.

Remote control, cloud synchronization, cloud alerts, account updates,
cloud analytics, and cloud lifecycle services may stop or become delayed
until Internet access is restored.

**ZH-TW  
**Internet
不可用指本地環境無外部網路存取。在此條件下，若本地網路、Hub、Gate、ArtiOS
與所需設備可用，本地網路功能可繼續運作。

遠端控制、雲端同步、雲端警報、帳號更新、雲端分析與雲端生命週期服務可能停止或延遲，直到
Internet 存取恢復。

## **14.6 Hub Unavailable**

## **14.6 Hub 不可用**

**EN  
**Hub unavailable means Artibird Hub cannot provide local runtime
services. In this condition, local runtime functions such as scene
execution, automation processing, MQTT services, local state management,
web services, or command routing may stop or degrade.

Some device-level behavior may continue depending on the device type,
firmware behavior, local direct control capability, power condition, and
product design. Any claim regarding behavior without Hub must be
validated.

**ZH-TW  
**Hub 不可用指 Artibird Hub 無法提供本地 Runtime
服務。在此條件下，場景執行、自動化處理、MQTT 服務、本地狀態管理、Web
服務或指令路由等本地 Runtime 功能可能停止或降級。

依設備類型、韌體行為、本地直接控制能力、供電條件與產品設計不同，部分設備級行為可能繼續。任何關於無
Hub 條件下行為的宣稱都必須完成驗證。

## **English Table**

| **Hub-Dependent Function** | **Behavior When Hub Is Unavailable** |
|----|----|
| Local scene execution | May stop or degrade |
| Local automation | May stop or degrade |
| Local MQTT service | Unavailable if hosted on Hub |
| Local state management | May stop or degrade |
| App local access through Hub | Unavailable or degraded |
| Gate-to-runtime routing | May stop or degrade |
| Device direct behavior | Depends on product type and firmware design |
| Physical control | May continue where device supports it directly |

## **中文表**

| **Hub 依賴功能**       | **Hub 不可用時行為**     |
|------------------------|--------------------------|
| 本地場景執行           | 可能停止或降級           |
| 本地自動化             | 可能停止或降級           |
| 本地 MQTT 服務         | 若由 Hub 承載，則不可用  |
| 本地狀態管理           | 可能停止或降級           |
| 經 Hub 的 App 本地存取 | 不可用或降級             |
| Gate 至 Runtime 路由   | 可能停止或降級           |
| 設備直接行為           | 取決於產品類型與韌體設計 |
| 實體控制               | 設備直接支援時可繼續     |

## **14.7 Gate Unavailable**

## **14.7 Gate 不可用**

**EN  
**Gate unavailable means Artibird Gate cannot provide device access,
onboarding, gateway communication, or supported protocol access. In this
condition, devices that depend on Gate for communication may become
unreachable or degraded.

Devices that use other supported local communication paths may continue
depending on deployment configuration. Gate failure behavior must be
described according to device category, network path, protocol
dependency, and validation result.

**ZH-TW  
**Gate 不可用指 Artibird Gate
無法提供設備接入、設備入網、網關通訊或受支援協議接入。在此條件下，依賴
Gate 通訊的設備可能不可達或降級。

使用其他受支援本地通訊路徑的設備，可依部署配置繼續運作。Gate
故障行為必須依設備類別、網路路徑、協議依賴與驗證結果進行描述。

## **14.8 Device Offline**

## **14.8 設備離線**

**EN  
**Device offline means a supported device cannot be reached by the local
runtime system, cloud service, gateway, App, or supported communication
path. The system should mark the device as offline where supported.

Commands to an offline device may be rejected, queued, retried, or
delayed depending on implementation. Offline behavior depends on product
type, communication path, power condition, firmware behavior, and
runtime configuration.

**ZH-TW  
**設備離線指受支援設備無法被本地 Runtime 系統、雲端服務、網關、App
或受支援通訊路徑連接。在支援條件下，系統應將設備標記為離線。

發送至離線設備的指令可依實作方式被拒絕、排隊、重試或延遲。離線行為取決於產品類型、通訊路徑、供電條件、韌體行為與
Runtime 配置。

## **English Table**

| **Offline Device Behavior** | **Description** |
|----|----|
| Mark offline | Device status is updated to offline where supported |
| Reject command | Command is rejected when device cannot be reached |
| Queue command | Command is stored for later execution where supported |
| Retry command | System retries communication according to implementation |
| Degraded state | Device remains visible but with limited capability |
| Alert generation | Offline event may generate alert where enabled |
| Log record | Offline condition may be logged where supported |

## **中文表**

| **設備離線行為** | **說明**                         |
|------------------|----------------------------------|
| 標記離線         | 在支援條件下將設備狀態更新為離線 |
| 拒絕指令         | 設備不可達時拒絕指令             |
| 指令排隊         | 在支援條件下儲存指令以待後續執行 |
| 重試指令         | 系統依實作方式重試通訊           |
| 降級狀態         | 設備仍可見，但能力受限           |
| 生成警報         | 啟用條件下離線事件可生成警報     |
| 日誌記錄         | 支援條件下可記錄離線狀態         |

Edge 本地 Runtime 行為應繼續。Edge 輔助功能可能停止、降級或不可用。

## **14.10 App Unavailable**

## **14.10 App 不可用**

**EN  
**App unavailable means Artibird App cannot access the system locally or
remotely. In this condition, supported physical controls, local scenes,
local automations, device-level behavior, and runtime services may
continue where required components are present.

App unavailable may prevent user configuration, remote control, device
management, scene editing, automation editing, status viewing, or
account management until App access is restored.

**ZH-TW  
**App 不可用指 Artibird App
無法在本地或遠端存取系統。在此條件下，當所需元件具備時，受支援實體控制、本地場景、本地自動化、設備級行為與
Runtime 服務可繼續運作。

App
不可用可能導致用戶無法進行配置、遠端控制、設備管理、場景編輯、自動化編輯、狀態查看或帳號管理，直到
App 存取恢復。

## **14.11 Power Failure**

## **14.11 電力故障**

**EN  
**Power failure defines how the system behaves when power is interrupted
for Hub, Gate, devices, local network equipment, Edge, or related
infrastructure.

Recovery depends on device power design, Hub startup behavior, Gate
startup behavior, network restoration, firmware state recovery, retained
configuration, and local runtime recovery. Battery-powered devices may
behave differently from mains-powered devices.

**ZH-TW  
**電力故障定義 Hub、Gate、設備、本地網路設備、Edge
或相關基礎設施供電中斷時，系統如何運作。

恢復取決於設備供電設計、Hub 啟動行為、Gate
啟動行為、網路恢復、韌體狀態恢復、保留配置與本地 Runtime
恢復。電池供電設備與市電供電設備可能具有不同表現。

## **English Table**

| **Power Failure Area** | **Expected Consideration** |
|----|----|
| Hub power failure | Local runtime may stop until Hub restarts |
| Gate power failure | Device access through Gate may stop until Gate restarts |
| Device power failure | Device becomes unavailable until power is restored |
| Network equipment power failure | Local and cloud communication may stop |
| Edge power failure | Edge-assisted functions become unavailable |
| Battery-powered device | May continue depending on battery condition |
| Mains-powered device | Depends on power restoration and firmware recovery |
| Configuration recovery | Depends on retained configuration and startup behavior |

## **中文表**

| **電力故障區域** | **預期考量**                                   |
|------------------|------------------------------------------------|
| Hub 斷電         | 本地 Runtime 可能停止，直到 Hub 重新啟動       |
| Gate 斷電        | 經 Gate 的設備接入可能停止，直到 Gate 重新啟動 |
| 設備斷電         | 設備不可用，直到電力恢復                       |
| 網路設備斷電     | 本地與雲端通訊可能停止                         |
| Edge 斷電        | Edge 輔助功能不可用                            |
| 電池供電設備     | 依電池狀態可繼續                               |
| 市電供電設備     | 取決於電力恢復與韌體恢復行為                   |
| 配置恢復         | 取決於保留配置與啟動行為                       |

## **14.12 Network Congestion**

## **14.12 網路擁塞**

**EN  
**Network congestion means local or cloud communication quality is
reduced by high traffic, interference, poor signal quality, message
overload, device count, gateway load, Hub load, or external network
conditions.

In network congestion conditions, system behavior may become delayed,
degraded, or temporarily unavailable. Capacity limits, message priority,
retry behavior, timeout behavior, and degradation rules should apply
where implemented.

**ZH-TW  
**網路擁塞指本地或雲端通訊品質因高流量、干擾、信號品質差、消息過載、設備數量、網關負載、Hub
負載或外部網路條件而降低。

在網路擁塞條件下，系統行為可能延遲、降級或暫時不可用。在已實作條件下，應適用容量限制、消息優先級、重試行為、超時行為與降級規則。

## **14.13 Degraded Operation**

## **14.13 降級運作**

**EN  
**Degraded operation means the system remains partially functional while
one or more components, services, or communication paths are unavailable
or limited.

A degraded state should be visible where supported. The system should
identify which capabilities remain available, which capabilities are
unavailable, and which capabilities may recover when the affected
component or service becomes available again.

**ZH-TW  
**降級運作指在一個或多個組件、服務或通訊路徑不可用或受限時，系統仍保持部分功能可用。

在支援條件下，降級狀態應可見。系統應識別哪些能力仍可用、哪些能力不可用，以及受影響組件或服務恢復後哪些能力可恢復。

## **English Table**

| **Degraded Condition** | **Available Behavior** | **Unavailable / Degraded Behavior** |
|----|----|----|
| Cloud unavailable | Local control, local scenes, local automation where supported | Remote access, cloud sync, cloud alerts |
| Edge unavailable | Non-edge runtime behavior | AI-assisted interpretation and optimization |
| Gate unavailable | Devices not dependent on Gate may continue | Gate-dependent device access |
| Hub unavailable | Device-level behavior where supported | Local runtime, scenes, automation |
| App unavailable | Physical controls and local automation where supported | User configuration and App control |
| Internet unavailable | Local network behavior where supported | Remote services and cloud sync |

## **中文表**

| **降級條件** | **可用行為** | **不可用 / 降級行為** |
|----|----|----|
| 雲端不可用 | 支援條件下本地控制、本地場景、本地自動化 | 遠端存取、雲端同步、雲端警報 |
| Edge 不可用 | 非 Edge Runtime 行為 | AI 輔助解讀與優化 |
| Gate 不可用 | 不依賴 Gate 的設備可繼續 | 依賴 Gate 的設備接入 |
| Hub 不可用 | 支援條件下設備級行為 | 本地 Runtime、場景、自動化 |
| App 不可用 | 支援條件下實體控制與本地自動化 | 用戶配置與 App 控制 |
| Internet 不可用 | 支援條件下本地網路行為 | 遠端服務與雲端同步 |

## **14.14 Recovery Behavior**

## **14.14 恢復行為**

**EN  
**Recovery behavior defines how the system returns from failure or
degraded state to normal operation. Recovery may include reconnecting
devices, restarting services, restoring state, synchronizing delayed
data, retrying commands, updating health status, clearing alerts, or
resuming cloud and edge participation.

Recovery behavior depends on the affected component, failure duration,
retained configuration, last known state, network condition, and
implementation design.

**ZH-TW  
**恢復行為定義系統如何從故障或降級狀態回到正常運作。恢復可包括重新連接設備、重啟服務、恢復狀態、同步延遲數據、重試指令、更新健康狀態、清除警報，或恢復雲端與
Edge 參與。

恢復行為取決於受影響組件、故障持續時間、保留配置、最後已知狀態、網路條件與實作設計。

## **English Table**

| **Recovery Item** | **Description** |
|----|----|
| Device reconnection | Device reconnects after network or power recovery |
| Hub service restart | Local runtime services restart after Hub recovery |
| Gate reconnection | Gate restores device access path |
| Cloud resynchronization | Delayed data sync resumes when cloud is available |
| Edge reconnection | Edge-assisted functions resume when Edge is available |
| State restoration | Last known or retained state is restored where supported |
| Command retry | Pending commands may retry where supported |
| Alert clearing | Alerts may clear after recovery condition is verified |

## **中文表**

| **恢復項目**  | **說明**                         |
|---------------|----------------------------------|
| 設備重新連接  | 網路或電力恢復後設備重新連接     |
| Hub 服務重啟  | Hub 恢復後本地 Runtime 服務重啟  |
| Gate 重新連接 | Gate 恢復設備接入路徑            |
| 雲端重新同步  | 雲端可用後延遲數據同步恢復       |
| Edge 重新連接 | Edge 可用後 Edge 輔助功能恢復    |
| 狀態恢復      | 支援條件下恢復最後已知或保留狀態 |
| 指令重試      | 支援條件下待處理指令可重試       |
| 警報清除      | 恢復條件確認後警報可清除         |

## **14.15 Reliability Boundary**

## **14.15 可靠性邊界**

**EN  
**Reliability and failure behavior must be described according to
supported components, product type, hardware version, firmware version,
software version, local network condition, cloud availability, edge
deployment status, power condition, device count, deployment model, and
validation evidence.

Reliability definitions do not imply uninterrupted operation, guaranteed
cloud availability, universal offline operation, unlimited capacity, or
failure-free behavior under all conditions.

**ZH-TW  
**可靠性與故障行為必須依據受支援組件、產品類型、硬體版本、韌體版本、軟體版本、本地網路條件、雲端可用性、Edge
部署狀態、供電條件、設備數量、部署模型與驗證證據進行描述。

可靠性定義不表示不間斷運作、不表示保證雲端可用、不表示通用離線運作、不表示無限制容量，也不表示所有條件下無故障行為。

## **14.16 Validation Requirements**

## **14.16 驗證要求**

**EN  
**Any reliability claim involving cloud failure, Internet failure, Hub
failure, Gate failure, device offline behavior, Edge failure, App
unavailability, power failure, network congestion, degraded operation,
or recovery behavior must be validated under defined conditions.

Validation should include product type, hardware version, firmware
version, software version, network topology, device count, power
