# **Chapter 5 — Device Coordination Architecture**

# **第五章｜設備協同架構**

---

## **5.1 Node Abstraction｜節點抽象**

**EN**  
In the OceanAI system, all devices are abstracted as functional nodes within a unified coordination model. Rather than being defined by hardware type, nodes are defined by their roles in sensing, interaction, or execution.

This abstraction allows heterogeneous devices to participate in a consistent system behavior regardless of physical differences.

**ZH-TW**  
在 OceanAI 系統中，所有設備被抽象為統一協同模型中的功能節點。節點不再依據硬體類型定義，而是依其在感知、交互或執行中的角色來界定。

此抽象使不同類型的設備能在一致的系統行為中協同運作，而不受物理差異限制。

---

## **5.2 Node Types｜節點類型**

**EN**  
The system defines several core node types:

- **Sense Node** — captures environmental data such as temperature, light, or motion
    
- **Control Node** — provides user interaction interfaces (e.g., panels, switches, mobile apps)
    
- **Execution Node** — performs physical actions such as switching, dimming, or actuation
    
- **Coordination Node** — manages system-level logic and orchestration (e.g., Hub, Edge)
    

Each node type contributes a specific function to the overall system behavior.

**ZH-TW**  
系統定義以下核心節點類型：

- **感知節點（Sense Node）** — 採集環境資料，如溫度、光照或動作
    
- **控制節點（Control Node）** — 提供使用者交互介面（如面板、開關、App）
    
- **執行節點（Execution Node）** — 執行物理行為，如開關、調光或驅動
    
- **協同節點（Coordination Node）** — 負責系統邏輯與協同（如 Hub、Edge）
    

各節點類型在整體系統行為中承擔不同功能。

---

## **5.3 State-Based Coordination｜基於狀態的協同**

**EN**  
Coordination in OceanAI is driven by system state rather than direct device-to-device commands.

- Nodes publish their current state
    
- The system maintains a shared state model
    
- Actions are triggered based on state conditions
    

This approach ensures consistent behavior and eliminates tight coupling between devices.

Within each runtime island, device coordination is mediated through ArtiOS Runtime. Device states and events enter the ArtiOS Runtime layer, where rules, scenes, automation, and logs determine how coordinated behavior is executed.

**ZH-TW**  
OceanAI 的協同基於系統狀態，而非設備之間的直接控制：

- 節點發布自身狀態
    
- 系統維護共享狀態模型
    
- 行為根據狀態條件觸發
    

此方式確保系統行為一致，並避免設備間的緊耦合。

在每座 Runtime 島嶼內部，設備協同透過 ArtiOS Runtime 承載。設備狀態與事件進入 ArtiOS Runtime 層，再由規則、場景、自動化與日誌決定協同行為如何執行。

---

## **5.4 Event-Driven Interaction｜事件驅動交互**

**EN**  
System behavior is triggered by events generated from user actions or environmental changes.

- Events are captured at node level
    
- Events propagate through the coordination layer
    
- System logic determines resulting actions
    

Event-driven interaction allows the system to respond dynamically to changing conditions.

**ZH-TW**  
系統行為由事件驅動，事件來自使用者操作或環境變化：

- 事件在節點層被捕捉
    
- 事件透過協同層傳播
    
- 系統邏輯決定對應行為
    

事件驅動使系統能動態回應變化情境。

---

## **5.5 Orchestration Model｜編排模型**

**EN**  
OceanAI uses a centralized coordination model at the logical level while maintaining distributed execution.

- Coordination logic is carried locally by Hub + ArtiOS Runtime and regionally by Edge coordination
    
- Decisions are made based on system state and rules
    
- Actions are dispatched to relevant execution nodes
    

This model balances system control and execution efficiency.

**ZH-TW**  
OceanAI 在邏輯層採用集中協同，在執行層保持分散：

- 協同邏輯在本地由 Hub + ArtiOS Runtime 承載，在區域層由 Edge 協同承載
    
- 決策依據系統狀態與規則產生
    
- 行為分發至相關執行節點
    

該模型平衡系統控制能力與執行效率。

---

## **5.6 Cross-Protocol Coordination｜跨協議協同**

**EN**  
Coordination extends across devices using different communication protocols.

**ZH-TW**

## **6.1 Data Flow Overview｜資料流總覽**

**EN**  
The OceanAI data flow architecture defines how information is generated, transmitted, processed, and consumed across the system. Data is treated as a continuous stream of state and events that flows through multiple layers without central interruption.

The system maintains coherence through structured data models and synchronized state propagation.

In local runtime environments, ArtiOS Runtime acts as the primary local point where device state, events, context, and runtime logs are organized before synchronization with Edge or Cloud layers.

**ZH-TW**  
OceanAI 資料流架構描述資訊在系統中如何產生、傳輸、處理與使用。資料以狀態與事件的形式持續流動，跨越多個層級而不依賴中心中斷。

系統透過結構化資料模型與狀態同步機制維持整體一致性。

在本地 Runtime 環境中，ArtiOS Runtime 是整理設備狀態、事件、情境與 Runtime 日誌的主要本地位置，之後再與 Edge 或 Cloud 層同步。

**EN**  
Data originates from multiple sources within the system:

All data is normalized into a unified representation before entering the system flow.

**ZH-TW**

## **7.2 Rule-Based Intelligence｜規則式智慧**

**EN**  
Rule-based intelligence is the foundation of OceanAI's automation logic.

Rules define clear relationships between conditions and actions. They allow users, installers, administrators, and system operators to create predictable behaviors inside a home, classroom, building, or campus.

Rules are executed locally through ArtiOS Runtime on Artibird Hub (see Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層).

A basic rule follows this structure:

```text
IF condition is true,
THEN execute action,
WITH constraints and permissions.
```

Examples include:

| Rule Type | Example |
| --- | --- |
| Time-based rule | Turn on hallway lights at 7:00 PM |
| Sensor-based rule | If motion is detected, activate lighting |
| State-based rule | If window is open, reduce HVAC output |
| Scene-based rule | If teaching mode starts, adjust lights and air-conditioning |
| Safety rule | If abnormal power state is detected, trigger alert |

Rule-based intelligence is deterministic. It provides stability, transparency, and basic automation reliability.

**ZH-TW**  
規則式智慧是 OceanAI 自動化邏輯的基礎。

規則定義條件與動作之間的明確關係。它讓使用者、安裝人員、管理者與系統運維者，可以在住宅、教室、樓宇或校園中建立可預期的行為。

規則透過 Artibird Hub 上的 ArtiOS Runtime 在本地執行（詳見 Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層）。

基本規則結構如下：

```text
如果條件成立，
則執行動作，
並遵循限制與權限。
```

示例包括：

| 規則類型 | 示例 |
| --- | --- |
| 時間規則 | 晚上 7:00 開啟走廊燈 |
| 感測規則 | 偵測到人在，啟動照明 |
| 狀態規則 | 窗戶打開時，降低空調輸出 |
| 場景規則 | 啟動教學模式時，調整照明與空調 |
| 安全規則 | 偵測到異常電力狀態時，觸發告警 |

規則式智慧是確定性的。它提供穩定性、透明性與基礎自動化可靠性。

**EN**  
When rules are executed locally, ArtiOS Runtime carries the rule state, trigger conditions, constraints, permissions, execution result, and runtime log. This makes local automation predictable, traceable, and resilient when cloud connectivity is limited.

**ZH-TW**  
當規則在本地執行時，ArtiOS Runtime 承載規則狀態、觸發條件、限制、權限、執行結果與 Runtime 日誌。這讓本地自動化在雲端連線受限時，仍保持可預期、可追蹤與具備韌性。

## **7.3 Context Intelligence｜情境智慧**

**EN**  
Context intelligence allows OceanAI to understand the meaning behind device signals.

A single signal is often not enough. Motion does not always mean occupancy. Brightness alone does not determine whether lights should change. Temperature alone does not define comfort. OceanAI combines multiple signals to understand spatial context.

Events from devices and user actions serve as triggers feeding context evaluation, connecting to the event-driven model defined elsewhere in this book.

Common context inputs include:

| Context Input | Meaning |
| --- | --- |
| Presence | Whether people are present |
| Time | Current time, schedule, routine, school period |
| Light | Brightness level and lighting condition |
| Climate | Temperature, humidity, air quality |
| Device State | Current device status, power, connection, mode |
| User Intent | App input, panel action, scene selection |
| Space Type | Home, classroom, office, corridor, campus zone |

Context intelligence turns raw data into operational meaning.

**ZH-TW**  
情境智慧讓 OceanAI 能夠理解設備訊號背後的意義。

單一訊號通常不足以形成可靠判斷。有動作不一定代表持續有人。亮度本身不一定決定是否需要調光。溫度本身也不等於舒適度。OceanAI 會綜合多個訊號理解空間情境。

來自設備與使用者動作的事件，作為觸發來源驅動情境評估，與全書所定義的事件驅動模型相連。

常見情境輸入包括：

| 情境輸入 | 含義 |
| --- | --- |
| Presence｜人在 | 是否有人存在 |
| Time｜時間 | 當前時間、日程、慣例、課表 |
| Light｜光照 | 亮度水平與照明狀態 |
| Climate｜氣候 | 溫度、濕度、空氣品質 |
| Device State｜設備狀態 | 當前設備狀態、電力、連線、模式 |
| User Intent｜使用者意圖 | App 輸入、面板操作、場景選擇 |
| Space Type｜空間類型 | 住宅、教室、辦公室、走廊、校園區域 |

情境智慧將原始資料轉化為可運行的意義。

**EN**  
Context intelligence provides meaning, but ArtiOS Runtime turns that meaning into local runtime behavior. Presence, time, light, climate, device state, and user intent are interpreted as context, then carried into scenes, rules, automation, and local execution through ArtiOS Runtime.

**ZH-TW**

## **9.1 Control Model Overview｜控制模型總覽**

**EN**  
The OceanAI control architecture defines how user intent enters the system and is transformed into coordinated actions. Control is not treated as direct device manipulation but as interaction with a system that interprets intent based on context and state.

Control interfaces read local runtime state, send user intent, and display execution feedback (see Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層). Intelligence interpretation occurs in Chapter 7 — Intelligence Layer｜第七章｜智慧層 before local execution.

The control model separates user interaction from execution, enabling flexible and scalable system behavior.

**ZH-TW**  
OceanAI 控制架構描述使用者意圖如何進入系統並轉化為協同行為。控制不再是對單一設備的直接操作，而是與一個能根據語境與狀態理解意圖的系統進行交互。

控制界面讀取本地 Runtime 狀態、送入使用者意圖，並顯示本地執行回饋（詳見 Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層）。智慧理解發生於 Chapter 7 — Intelligence Layer｜第七章｜智慧層，本地執行則由本地 Runtime 層承載。

控制模型將使用者交互與實際執行分離，使系統具備靈活與可擴展性。

## **9.10 Chapter Summary｜本章總結**

**EN**  
The OceanAI control architecture defines how human intent enters the runtime without reducing control to direct device manipulation.

App, Panel, switch, and voice interfaces act as intent entry points. Intent is abstracted, interpreted with context, and routed through a control flow that reads ArtiOS Runtime runtime state and returns local execution feedback. Local-first control, context awareness, and multi-interface consistency keep interaction predictable across environments.

Control sits above ArtiOS Runtime as the human interaction layer. Intelligence interpretation occurs in Chapter 7 — Intelligence Layer｜第七章｜智慧層; local execution is carried by ArtiOS Runtime in Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層.

**OceanAI's control architecture turns user intent into coordinated spatial behavior through context-aware, multi-interface interaction above local ArtiOS Runtime execution.**

**ZH-TW**  
OceanAI 控制架構定義人類意圖如何進入 Runtime，而不把控制降格為對設備的直接操作。

App、Panel、開關與語音界面作為意圖入口。意圖被抽象化、依情境理解，並沿控制流程讀取 ArtiOS Runtime 狀態、回傳本地執行回饋。本地優先控制、情境感知與多介面一致性，使交互在不同環境中保持可預期。

控制位於 ArtiOS Runtime 之上，作為人機互動層。智慧理解發生於 Chapter 7 — Intelligence Layer｜第七章｜智慧層；本地執行則由 Chapter 8 — ArtiOS Runtime Layer｜第八章｜ArtiOS Runtime 層 的 ArtiOS Runtime 承載。

**OceanAI 的控制架構，透過位於 ArtiOS Runtime 本地執行之上、具情境感知且多介面一致的人機互動，將使用者意圖轉化為協同的空間行為。**
