---

## **1.1 System Boundary｜系統邊界**

**EN**  
OceanAI operates as a system layer spanning cloud infrastructure, edge computation, local coordination, and device execution. It establishes a continuous operational domain in which all system components interact under a unified model.

**ZH-TW**  
OceanAI 作為一個系統層，橫跨雲端基礎設施、邊緣運算、本地協同與設備執行，形成一個連續運作的系統域，使所有組件在統一模型下交互運作。

---

## **1.2 System Composition｜系統構成**

**EN**  
The OceanAI system is composed of six coordinated architecture layers:

- **Cloud Intelligence Layer** — system-wide intelligence, data services, and remote coordination
    
- **Edge Coordination Layer** — near-environment coordination and regional decision support
    
- **Hub Local Runtime Layer** — local runtime host; ArtiOS Runtime runs primarily through Hub
    
- **Gateway Access Layer** — network access, protocol bridging, and mesh coordination
    
- **Synchronization Layer** — IR / RF and legacy-appliance synchronization
    
- **Device Execution Layer** — sensing, control, actuation, and physical feedback
    

These layers operate as an integrated structure rather than independent subsystems.

**ZH-TW**  
OceanAI 系統由六個協同架構層構成：

- **雲端智慧層（Cloud Intelligence Layer）** — 系統級智慧、資料服務與遠端協同
    
- **邊緣協同層（Edge Coordination Layer）** — 靠近環境的協同與區域決策支援
    
- **中樞本地層（Hub Local Runtime Layer）** — 本地 Runtime 承載；
    
- **網關接入層（Gateway Access Layer）** — 網路接入、協議橋接與 Mesh 協同
    
- **同步接入層（Synchronization Layer）** — IR / RF 與傳統家電同步
    
- **設備執行層（Device Execution Layer）** — 感知、控制、執行與物理回饋
    

各層並非獨立運作，而是構成一個整合的系統結構。

---

## **1.3 System Roles｜系統角色**

**EN**  
Within this structure, OceanAI governs system logic, coordination, and intelligence, while Artibird implements these roles through physical devices. Each device functions as a node within the system, participating in sensing, interaction, or execution.

System behavior emerges from the interaction of these roles rather than from any individual component.

**ZH-TW**  
在此結構中，OceanAI 負責系統邏輯、協同機制與智慧運作，而 Artibird 則透過實體設備實現這些角色。每個設備作為系統中的節點，參與感知、交互或執行。

系統行為來自各角色之間的協同，而非單一組件。

---

## **1.4 Coordination Model｜協同模型**

**EN**  
OceanAI operates on a coordinated system model based on state exchange and event-driven interaction:

- Devices continuously publish state information
    
- Events are generated from user actions or environmental changes
    
- The system processes events through coordination logic
    
- Actions are executed across nodes based on system state and context
    

This model enables consistent behavior across distributed components without centralized dependency.

**ZH-TW**  
OceanAI 採用基於狀態交換與事件驅動的協同模型運作：

- 設備持續發布狀態資訊
    
- 事件由使用者操作或環境變化產生
    
- 系統透過協同邏輯處理事件
    
- 根據系統狀態與情境，在各節點間執行行為
    

該模型使分散組件之間能在無集中依賴的情況下保持一致運作。

---

## **1.5 Operational Principles｜運作原則**

**EN**  
The system operates under the following principles:

- **Local-first execution** — core control logic is processed locally
    
- **Deterministic response** — system behavior remains predictable and stable
    
- **Cloud-extended capability** — cloud services enhance but do not sustain core operation
    
- **Protocol abstraction** — heterogeneous communication protocols operate under a unified model
    

These principles ensure system continuity, low latency, and scalability across deployment scenarios.

**ZH-TW**  
系統運作遵循以下原則：

- **本地優先（Local-first）** — 核心控制邏輯在本地執行
    
- **確定性回應（Deterministic）** — 系統行為保持可預測與穩定
    
- **雲端擴展（Cloud-extended）** — 雲端提升能力，但不依賴其維持運作
    
- **協議抽象（Protocol abstraction）** — 異質通訊協議在統一模型下運作
    

這些原則確保系統具備低延遲、連續性與可擴展性。

## **2.0 Chapter Purpose｜本章目的**

**EN**  
This chapter defines the core architecture of OceanAI.

OceanAI is designed as a distributed runtime system for intelligent spaces. **OceanAI Runtime (ORT)** is the system-level runtime architecture that coordinates this continuous operation across cloud, edge, local runtime, and device layers. Its architecture connects cloud intelligence, edge coordination, local runtime, gateway access, device execution, and legacy-device synchronization into one continuously operating system.

The purpose of this chapter is to explain how OceanAI moves from system definition to operational architecture: how intelligence is distributed, how environments remain locally responsive, and how different spaces coordinate as part of a larger runtime network.

**ZH-TW**  
本章定義 OceanAI 的核心架構。

	
>OceanAI 被設計為面向智慧空間的分布式 Runtime 系統。**OceanAI Runtime（ORT）** 是協調雲端、邊緣、本地 Runtime 與設備層之間持續運行的系統級 Artibird Runtime（ART） 架構。它將雲端智慧、邊緣協同、本地運行、網關接入、設備執行與傳統設備同步，連接為一個持續運行的系統。

本章的目的，是說明 OceanAI 如何從系統定義進入可運行架構：智慧如何分布，空間如何保持本地回應能力，不同環境如何成為更大 Runtime 網路的一部分。

## **2.1 Core Architecture Principle｜核心架構原則**

**EN**  
OceanAI follows one architectural principle:

> **Local autonomy. System coordination. Adaptive intelligence.**

Each space must be able to operate locally.  
Each device must be able to participate in a larger system.  
Each layer must contribute to intelligence without depending on a single point of control.

This principle allows OceanAI to support homes, classrooms, buildings, campuses, and distributed environments where reliability, continuity, and coordination matter.

**ZH-TW**  
OceanAI 遵循一個核心架構原則：

> **本地自治  系統協同  自適應智慧**


每個空間都需要具備本地運行能力。  
每個設備都需要能夠參與更大的系統。  
每一層都需要貢獻智慧能力，並共同支撐整體協同。

這一原則讓 OceanAI 可以支援住宅、教室、樓宇、校園與分布式環境，適用於重視可靠性、連續性與協同能力的真實場景。

**EN**  
OceanAI defines the spatial runtime architecture.  
ArtiOS Runtime makes this runtime locally executable inside Artibird environments.  
Artibird devices bring runtime intelligence into physical space.

This relationship keeps the architecture clear: Runtime remains the upper-level system method, while ArtiOS Runtime serves as the local runtime operating layer.

**ZH-TW**  
OceanAI 定義空間 Runtime 架構。  
ArtiOS Runtime 使這套 Runtime 能在 Artibird 環境中本地可執行。  
Artibird 設備將 Runtime 智慧帶入物理空間。

這一關係讓架構保持清晰：Runtime 仍是上位系統方法，ArtiOS Runtime 則是本地 Runtime 運行層。

## **2.2 Islands  Architecture｜群島式群集架構**

**EN**  
OceanAI adopts an **Islands + Swarm Runtime Architecture**.

In this model, each local environment operates as a runtime island. A room, floor, home, classroom, building, or campus zone can maintain its own sensing, control, rule execution, device state, and local response capability.

These runtime islands are connected through gateways, hubs, cloud services, and coordination protocols. Each island keeps local autonomy, while the system as a whole maintains continuous coordination.

Swarm intelligence defines how these islands and their internal devices cooperate. Sensors, switches, panels, plugs, relays, synchronizers, gateways, hubs, and visual nodes act as distributed participants in a larger runtime field.

OceanAI therefore creates intelligent spaces that are locally resilient, collectively aware, and continuously coordinated.

**ZH-TW**  
OceanAI 採用 **Islands Architecture｜群島式架構**。

在這個模型中，每一個本地環境都可以作為一座 Runtime 島嶼獨立運行。一個房間、一層樓、一個家庭、一間教室、一棟建築，或一個校園區域，都可以保有自己的感知、控制、規則執行、設備狀態與本地回應能力。

這些 Runtime 島嶼透過網關、中樞、雲端服務與協同協議保持連接。每座島嶼保有本地自治能力，整個系統則維持持續協同。

群集智能定義這些島嶼與內部設備如何共同運作。感測器、開關、面板、插座、繼電器、同步器、網關、中樞與視覺節點，都是更大 Runtime 場中的分布式參與者。

因此，OceanAI 建立的是具備本地韌性、群體感知與持續協同能力的智慧空間。

**EN**  
In the Islands + Swarm model, ArtiOS Runtime is the local runtime layer that makes each island executable. Without ArtiOS Runtime, an island remains an architectural concept. With ArtiOS Runtime, each local environment can maintain its own state, rules, scenes, events, logs, and offline fallback.



**ZH-TW**  
在群島式群集模型中，ArtiOS Runtime 是讓每座島嶼具備可執行能力的本地 Runtime 實現層。沒有 ArtiOS Runtime，島嶼仍然只是架構概念；有了 ArtiOS Runtime，每個本地環境才能維持自己的狀態、規則、場景、事件、日誌與離線備援。

它的角色是觀察更大的系統，支援多場域協同，維護使用者與設備關係，並提供本地層難以完全承擔的高階智慧能力。

## **2.5 Edge Coordination Layer｜邊緣協同層**

**EN**  
The Edge Coordination Layer connects cloud intelligence with local runtime execution.

It reduces latency, improves resilience, and allows important decisions to happen closer to the physical environment. In real-world spaces, many actions require immediate response: lighting changes, occupancy detection, air-conditioning control, scene execution, alarm response, and device-state synchronization.

The edge layer allows OceanAI to keep intelligence close to the environment.

**ZH-TW**  
邊緣協同層連接雲端智慧與本地 Runtime 執行。

它降低延遲、提升韌性，讓重要判斷能夠靠近真實物理環境完成。在真實空間中，許多動作都需要即時回應，例如照明變化、人在感測、空調控制、場景執行、告警回應與設備狀態同步。

邊緣層讓 OceanAI 的智慧更靠近空間本身。

## **2.7 Gateway Access Layer｜網關接入層**

**EN**  
The Gateway Access Layer connects local devices and networks into the OceanAI runtime system.

The Gate provides network access, protocol bridging, mesh coordination, local routing, and secure communication between devices, hubs, and cloud services. It allows devices with different communication methods to join the same coordinated environment.

In the Islands + Swarm model, the Gate acts as the bridge between runtime islands and device networks.

**ZH-TW**  
網關接入層將本地設備與網路接入 OceanAI Runtime 系統。

Gate 提供網路接入、協議橋接、Mesh 協同、本地路由，以及設備、中樞與雲端服務之間的安全通訊。它讓使用不同通訊方式的設備，可以加入同一個協同環境。

在群島式群集模型中，Gate 是 Runtime 島嶼與設備網路之間的橋。

## **2.8 Synchronization Layer｜同步接入層**

**EN**  
The Synchronization Layer allows legacy appliances and non-native devices to participate in the OceanAI system.

Through Artibird Sync, IR, RF, and traditional remote-controlled appliances can be synchronized into the runtime environment. This allows OceanAI to extend intelligence beyond newly installed smart devices and include existing appliances in the coordinated system.

In the Islands + Swarm model, Sync acts as a legacy-device adapter.

**ZH-TW**  
同步接入層讓傳統家電與非原生智慧設備也能參與 OceanAI 系統。

透過 Artibird Sync，IR、RF 與傳統遙控家電可以被同步到 Runtime 環境中。這讓 OceanAI 的智慧能力可以延伸到新安裝的智慧設備之外，也能將既有家電納入協同系統。

在群島式群集模型中，Sync 是傳統設備接入器。

## **2.9 Device Execution Layer｜設備執行層**

**EN**  
The Device Execution Layer is where system intelligence becomes physical action.

Switches, panels, plugs, relays, sensors, visual nodes, and environmental devices perform the actual sensing and execution inside the space. They detect changes, receive commands, report state, and trigger local or system-level responses.

In the OceanAI architecture, devices are runtime participants. Each device contributes to the environment's awareness, response, and coordination.

**ZH-TW**  
設備執行層是系統智慧轉化為物理動作的地方。

開關、面板、插座、繼電器、感測器、視覺節點與環境設備，在空間中完成實際感知與執行。它們偵測變化、接收指令、回報狀態，並觸發本地或系統級回應。

在 OceanAI 架構中，設備是 Runtime 參與者。每個設備都為環境的感知、回應與協同提供能力。

## **3.1 System Model and Layered Runtime Structure｜系統模型與分層結構**

**EN**

This chapter defines the system model of OceanAI—the structural vocabulary through which cloud intelligence, edge coordination, local runtime execution, and physical device systems relate as one spatial runtime architecture.

OceanAI is not a catalog of products. It is a system-level intelligence architecture that organizes how environments sense, interpret, coordinate, act, and learn across distributed layers. The system model answers three questions that every deployment must resolve: where intelligence lives, where execution authority resides, and how responsibility crosses domain boundaries without collapsing into a single point of failure.

At the highest level, OceanAI follows a layered spatial runtime structure:

```text
OceanAI Cloud Intelligence
        ↓
OceanAI Edge Coordination
        ↓
ArtiOS Runtime Local Runtime Boundary
        ↓
Artibird Device System
        ↓
Physical Space
```

Each layer has a distinct operational role. **OceanAI Runtime (ORT)** defines the system-level architecture of continuous operation—the rules, flows, and coordination logic that govern how intelligence moves across the entire system. **ArtiOS Runtime (ART)** is the local implementation of ORT inside each Artibird environment. ART is not a separate architecture; it is the executable boundary that makes ORT locally real.

This distinction matters for spatial sovereignty. A physical environment must retain the ability to operate when upstream services are delayed, degraded, or unavailable. The layered model therefore separates *system-wide intelligence* from *local execution authority*. Cloud and Edge can observe, coordinate, and optimize; ArtiOS Runtime preserves the right and capability to execute critical behavior locally.

The model also clarifies product placement. **Artibird** is the physical device ecosystem that realizes OceanAI in deployable hardware. Devices are runtime participants—not isolated endpoints—connected through ArtiOS Runtime state, event flow, local rules, and coordination logic. Physical space is the outermost layer: the domain where runtime intelligence becomes tangible action.

Understanding this layered structure is prerequisite to reading the three-domain model, the ArtiOS Runtime boundary, cross-domain flows, and Runtime island mapping that follow.

**ZH-TW**

本章定義 OceanAI 的系統模型——這是一套結構性語彙，用來描述雲端智慧、邊緣協同、本地 Runtime 執行與實體設備系統，如何共同構成一套空間 Runtime 架構。系統模型不是抽象概念圖，而是部署、開發與運維決策的結構性依據：它規定智慧應在何處產生、執行權威應在何處歸屬、責任應如何在域邊界之間流動而不坍縮為單點故障。

OceanAI 不是產品目錄，而是系統級智慧架構，組織環境如何在分布式層級之間完成感知、理解、協同、執行與學習。與傳統智慧家居以「設備連網」為中心的思路不同，OceanAI 以「空間 Runtime」為中心：設備是 Runtime 參與者，行為從協同中湧現，智慧是系統屬性而非單一產品功能。系統模型因此必須同時回答架構問題與落地問題——前者關乎各層如何分工，後者關乎 Artibird 硬體如何在真實空間中承載這套分工。

在最高層級上，OceanAI 採用分層式空間 Runtime 結構：

```text
OceanAI 雲端智慧
        ↓
OceanAI 邊緣協同
        ↓
ArtiOS Runtime 本地 Runtime 邊界
        ↓
Artibird 設備系統
        ↓
真實物理空間
```

每一層都有明確的運作角色。**OceanAI Runtime（ORT）** 定義持續運行的系統級架構——規則、流程與協同邏輯，治理智慧如何在整體系統中流動。ORT 是方法論層：它描述 Runtime 應如何組織、如何流動、如何保持連續性。**ArtiOS Runtime（ART）** 則是 ORT 在每個 Artibird 環境中的本地實現。ART 並非另一套獨立架構，而是使 ORT 在本地成為可執行現實的 Runtime 邊界。若將 ORT 比作系統級「憲法」，ART 便是各環境的「本地執法機構」——在統一規則下保有本地自治。

這一區分對空間主權至關重要。物理環境必須在上游服務延遲、降級或不可用時，仍保有持續運作的能力。分層模型因此將「系統級智慧」與「本地執行權威」分離：Cloud 與 Edge 可以觀察、協同與優化；ArtiOS Runtime 則保留在本地執行關鍵行為的權利與能力。空間主權不是政治隱喻，而是工程要求——照明、安防、氣候與緊急回應等物理行為，不能因雲端回應延遲而中斷。

模型也釐清產品定位。**Artibird** 是將 OceanAI 落地為可部署硬體的實體設備生態。設備是 Runtime 參與者，而非孤立端點——它們透過 ArtiOS Runtime 的狀態、事件流、本地規則與協同邏輯彼此連接。Hub 承載本地大腦，Gate 提供網路橋接，Sync 延伸傳統家電接入，感測器與執行器完成物理感知與動作。物理空間是最外層：Runtime 智慧轉化為可感知、可觸控、可回饋的動作域。

分層結構還揭示一項基本設計原則：系統行為應從協同中湧現，而非從單一控制點發出。當環境擴展到多房間、多樓層或多場域時，每一層都必須在自身職責範圍內保持自治，同時向上層提供可聚合的狀態與事件。Cloud 看見跨時間的模式與跨場域的關聯；Edge 看見跨區域的協同需求與低延遲決策；ArtiOS Runtime 看見本地連續性、規則執行與離線韌性；設備看見物理變化與人機互動。四種視角並存，構成完整的空間 Runtime 圖像。

從運維視角看，分層結構也定義了故障隔離邊界。Cloud 不可用時，本地環境應透過 ART 繼續運行；Edge 不可用時，各 Runtime island 應保持自治；單一設備故障時，不應拖垮整個 island 的連續性。這種隔離不是消極備援，而是主動的架構設計——每一層都預設可能降級，並在降級模式下仍保有核心能力。

理解這套分層結構，是閱讀後續三域模型、ArtiOS Runtime 邊界、跨域流程與 Runtime island 映射的前提。沒有清晰的層級邊界，協同將退化為對雲端的依賴；沒有本地 Runtime 邊界，智慧空間將在網路中斷時失去連續性。系統模型正是為避免這兩種極端而建立，使 OceanAI 能在全局協同與本地自治之間取得結構性平衡。

從設計方法論看，分層 Runtime 結構也定義了擴展路徑。新增設備時，只需在 Device 層接入並向 ART 註冊狀態與能力；新增場域時，只需部署新的 Runtime island 並在 Cloud 註冊身份關係；新增區域協同時，只需在 Edge 層定義跨 island 的協同策略。各層接口穩定，擴展不必重構整體架構。這正是系統級架構與產品堆疊的本質差異：OceanAI 提供可組合的 Runtime 語義，Artibird 提供可部署的物理節點，兩者在分層模型中對齊而非耦合。

分層模型還為測試與驗證提供邊界。每一層都可以獨立驗證其職責是否被正確實現：Cloud 服務是否正確聚合與下發策略，Edge 是否正確調度區域協同，ART 是否在離線條件下維持本地連續性，Device 是否正確發布狀態與執行指令。這種可分層驗證的特性，降低了大型部署的集成風險，使問題定位可以從最可能出錯的層級開始，而非在整體黑箱中盲目排查。

對比傳統智慧家居「App 控制設備」的線性模型，OceanAI 的分層 Runtime 結構將「控制」重新定義為「協同下的持續運行」。使用者觸發的不是單一指令，而是進入 Runtime 流程的意圖；設備回應的不是孤立動作，而是 island 內協同邏輯的執行結果。這種范式轉換，正是系統模型需要被明確定義的原因——沒有共同語彙，設計與部署將退回產品堆疊的老路，失去 OceanAI 作為系統架構的結構性價值。

## **3.2 Cloud, Edge, Device Domains｜三域職責**

**EN**

OceanAI organizes system responsibility into three major domains—Cloud, Edge, and Device—each with a distinct scope, latency profile, and authority boundary.

The **Cloud Domain** provides system-wide intelligence and long-term coordination. It manages accounts, organizations, sites, homes, devices, permissions, historical data, AI-assisted analysis, remote access, and multi-site coordination. Cloud services allow OceanAI to observe patterns across time and across environments—capabilities that require broader data scope, centralized governance, or remote reach.

Typical Cloud responsibilities include:

| Capability | Description |
| --- | --- |
| Identity and account management | Users, roles, organizations, homes, sites, and permissions |
| Device registry | Device identity, ownership, lifecycle, firmware, and status |
| Data services | Historical logs, runtime records, events, and environmental data |
| AI-assisted analysis | Pattern recognition, anomaly detection, prediction, and recommendations |
| Remote access | External control, monitoring, and management |
| Multi-site coordination | Coordination across homes, buildings, campuses, or distributed deployments |

The **Edge Coordination Domain** brings intelligence closer to the physical environment. It supports regional coordination, low-latency interpretation, local data aggregation, cross-room coordination, and near-environment decision support. Edge is especially important in buildings, campuses, schools, and multi-zone environments where multiple Runtime islands must operate together. Edge does not replace ArtiOS Runtime; it provides a broader regional intelligence layer while ART preserves local executability.

The **Device Domain** is where runtime intelligence becomes physical action. Artibird devices provide sensing, control, actuation, feedback, and human interaction. Devices include Hub, Gate, Sync, switches, panels, plugs, relays, sensors, vision nodes, and other physical runtime participants. In the OceanAI system model, devices are not isolated products—they are runtime participants connected through ArtiOS Runtime state, event flow, local rules, and coordination logic.

| Domain | Primary Role | Typical Scope |
| --- | --- | --- |
| Cloud | System-wide intelligence, learning, governance | Cross-site, long-term |
| Edge | Regional coordination, low-latency interpretation | Building, campus, multi-zone |
| Device | Sensing, control, actuation, physical feedback | Room, zone, appliance |

Cloud strengthens the system but must not become the sole dependency for runtime behavior. Critical local actions should remain executable through ArtiOS Runtime whenever possible—a principle that defines the relationship between domains rather than ranking them by importance.

**ZH-TW**

OceanAI 將系統責任組織為三個主要域——Cloud、Edge 與 Device——各自具有不同的職責範圍、延遲特性與權威邊界。三域不是三層堆疊，而是三種責任視野：Cloud 看見全局，Edge 看見區域，Device 看見物理。ArtiOS Runtime 作為本地執行錨點，將三域視野匯聚為可執行的本地行為。理解三域職責，是設計部署拓撲、劃分接口邊界與制定 local-first 策略的基礎。

**Cloud Domain（雲端域）** 提供系統級智慧與長期協同。它管理帳號、組織、場域、家庭、設備、權限、歷史資料、AI 輔助分析、遠端接入與多場域協同。雲端服務使 OceanAI 能跨時間、跨環境觀察模式——這些能力需要更大的資料範圍、集中治理或遠端觸達。Cloud 的價值在於「看得更廣、記得更久、學得更深」：它聚合多場域 Runtime 日誌，辨識長期模式，支援 AI 輔助優化，並提供遠端管理入口。但 Cloud 不應成為日常環境回應的唯一路徑——關鍵物理動作必須能在本地閉環完成。

典型 Cloud 職責包括：

| 能力 | 說明 |
| --- | --- |
| 身份與帳號管理 | 使用者、角色、組織、家庭、場域與權限 |
| 設備註冊 | 設備身份、歸屬、生命週期、韌體與狀態 |
| 資料服務 | 歷史日誌、Runtime 記錄、事件與環境資料 |
| AI 輔助分析 | 模式識別、異常檢測、預測與建議 |
| 遠端接入 | 外部控制、監測與管理 |
| 多場域協同 | 跨住宅、樓宇、校園或分布式部署的協同 |

**Edge Coordination Domain（邊緣協同域）** 讓智慧更靠近物理環境。它支援區域協同、低延遲理解、本地資料聚合、跨房間協同與靠近環境的決策支援。Edge 在樓宇、校園、學校與多區域環境中尤其重要——此時多個 Runtime island 需要共同運行，卻又不適合將所有決策上送 Cloud。Edge 填補 Cloud 與本地之間的區域空白：它比 Cloud 更近、比單一 island 更廣，擅長跨房間調度、區域情境理解與多 island 協同。Edge 不取代 ArtiOS Runtime；它提供更大範圍的區域智慧層，而 ART 維持本地可執行性。

**Device Domain（設備域）** 是 Runtime 智慧轉化為物理動作之處。Artibird 設備提供感知、控制、執行、回饋與人機互動。設備包括 Hub、Gate、Sync、開關、面板、插座、繼電器、感測器、視覺節點與其他物理 Runtime 參與者。在 OceanAI 系統模型中，設備不是孤立產品——它們是透過 ArtiOS Runtime 狀態、事件流、本地規則與協同邏輯連接的 Runtime 參與者。每一台設備都發布狀態、接收指令、觸發事件，並在協同邏輯下與其他節點共同形塑環境行為。

| 域 | 主要角色 | 典型範圍 |
| --- | --- | --- |
| Cloud | 系統級智慧、學習、治理 | 跨場域、長期 |
| Edge | 區域協同、低延遲理解 | 樓宇、校園、多區域 |
| Device | 感知、控制、執行、物理回饋 | 房間、區域、家電 |

三域之間的關係遵循一項核心原則：Cloud 強化系統，但不應成為 Runtime 行為的唯一依賴。關鍵本地動作應盡可能透過 ArtiOS Runtime 保持可執行。Cloud 擅長全局任務：跨場域身份管理、長期學習、歷史分析與遠端接入。Edge 擅長區域任務：跨房間協同、低延遲情境理解、多 Runtime island 調度。Device 擅長物理任務：環境感知、即時執行、人機互動與物理回饋。三域視野互補，共同構成完整的空間 Runtime 能力。

責任邊界表進一步釐清各域的優先歸屬：

| 責任 | 優先域 |
| --- | --- |
| 帳號、身份、權限 | Cloud |
| 長期學習與歷史分析 | Cloud |
| 遠端控制與多場域協同 | Cloud |
| 區域協同 | Edge |
| 跨房間或跨區域理解 | Edge |
| 物理感知與執行 | Device |
| 人機觸控輸入與本地回饋 | Device |

本地規則、場景執行、設備狀態、本地事件流與離線控制等職責，則歸屬 ArtiOS Runtime 邊界——這是 Cloud、Edge 與 Device 三域之間的本地 Runtime 中樞。三域模型以 ART 為錨點：Cloud 與 Edge 提供增強與協同，Device 提供物理觸達，ART 將三者匯聚為本地可執行的連續 Runtime。部署時應先問「這項能力屬於哪個域」，再問「它是否必須經過 ART 才能落地」——這兩個問題決定了接口設計與 local-first 策略的具體形態。三域職責的清晰劃分，也是避免「所有邏輯上雲」或「所有決策本地」兩種極端部署的結構性保障。

在接口設計實務中，三域之間的資料流也應遵循職責邊界。Cloud 向 ART 下發的是策略與配置，而非逐條執行指令；Edge 向 ART 提供的是區域語境與協同請求，而非取代本地決策；Device 向 ART 回報的是狀態與事件，而非等待遠端解析。這種單向職責流動，確保每一域都在自身邊界內運作，避免責任滲透導致的耦合與故障傳播。三域模型因此既是概念框架，也是接口契約的基礎。

當多個 Runtime island 共存於同一物理場域時，三域職責的協調尤為關鍵。Cloud 維護 island 之間的身份關係與全局策略；Edge 定義 island 之間的區域協同規則，如跨房間場景或樓層聯動；各 island 的 ART 則保持本地自治，只在收到合法協同請求時參與跨 island 動作。這種「本地自治、按需協同」的模式，是 OceanAI 區別于集中式樓宇自控系統的結構性特徵。三域職責的清晰劃分，使每個域都能在自身邊界內被獨立設計、測試與演進，而不必因某一域的變更而牽動整體重構。這種模組化責任結構，是 OceanAI 系統模型支持多規模、多場景部署的基礎能力。無論是單戶家庭還是大型園區，Cloud、Edge、Device 三域的語義與邊界保持不變，僅協同範圍與部署密度隨場景調整。三域模型是 OceanAI 系統架構中最穩定的結構單元之一。

## **3.3 ArtiOS Runtime Boundary｜ArtiOS Runtime 邊界**

**EN**

Between Edge coordination and device execution lies **ArtiOS Runtime (ART)**—the local runtime boundary where system intent becomes locally executable behavior.

In the OceanAI system model, ART is not an optional module. It is the layer that makes **OceanAI Runtime (ORT)** executable inside each Artibird environment. Without ART, ORT remains an architectural specification; with ART, each local environment maintains its own state, rules, scenes, events, device coordination, logs, and offline fallback.

ART sits at the intersection of three domains:

| Responsibility | Preferred Layer |
| --- | --- |
| Local rules and scene execution | ArtiOS Runtime |
| Device state and local event flow | ArtiOS Runtime |
| Offline control and degraded operation | ArtiOS Runtime |
| Human touch input and local feedback | Device / ArtiOS Runtime |
| Regional coordination | Edge |
| Long-term learning and remote access | Cloud |
| Physical sensing and actuation | Device |

**Hub**, **Gate**, and **Sync** are Artibird hardware products that ART coordinates as one local system:

- **Hub** acts as the primary local runtime host—the local brain of a Runtime island. Local runtime runs primarily through Hub.
- **Gate** provides gateway access, mesh coordination, and network bridging between runtime islands and device networks.
- **Sync** provides IR / RF synchronization, extending runtime reach to legacy appliances.

These products are not interchangeable. Hub is not Gate; Gate is not Sync. Each carries a distinct runtime role, unified by ArtiOS Runtime into a coherent local execution environment.

This separation prevents the system from becoming dependent on any single layer. Cloud can enhance intelligence; Edge can coordinate regions; ART preserves local continuity; devices continue sensing and executing. The ArtiOS Runtime boundary is where spatial sovereignty becomes operational—where a room, home, or building zone retains the authority to act even when upstream coordination is unavailable.

**ZH-TW**

Edge 協同與設備執行之間，存在 **ArtiOS Runtime（ART）**——本地 Runtime 邊界，系統意圖在此轉化為本地可執行行為。ART 是 OceanAI 系統模型中最關鍵的結構性節點：它既不是 Cloud 的下屬，也不是 Device 的上位，而是將 ORT 的系統級規則落地為本地連續 Runtime 的執行層。沒有 ART，三域模型將失去本地錨點，空間主權將無從談起。

在 OceanAI 系統模型中，ART 不是可選模組。它是使 **OceanAI Runtime（ORT）** 在每個 Artibird 環境中可執行的層級。ORT 定義持續運行的架構方法——狀態如何表示、事件如何傳播、指令如何執行、協同如何維持。ART 則在 Hub、Gate、Sync 與其他 Artibird 節點上，將這套方法轉化為可運行的本地程序。沒有 ART，ORT 仍只是架構規格；有了 ART，每個本地環境才能維持自己的狀態、規則、場景、事件、設備協同、日誌與離線備援。

ART 處於三域的交會點，承接 Cloud 與 Edge 無法或不應承担的本地職責：

| 責任 | 優先層級 |
| --- | --- |
| 本地規則與場景執行 | ArtiOS Runtime |
| 設備狀態與本地事件流 | ArtiOS Runtime |
| 離線控制與降級運行 | ArtiOS Runtime |
| 人機觸控輸入與本地回饋 | Device / ArtiOS Runtime |
| 區域協同 | Edge |
| 長期學習與遠端接入 | Cloud |
| 物理感知與執行 | Device |

**Hub**、**Gate** 與 **Sync** 是 Artibird 硬體產品；ART 將它們協同為一個本地系統，而非三個獨立設備的鬆散集合：

- **Hub** 作為主要本地 Runtime 承載體，是 Runtime island 的本地大腦。本地 Runtime 主要透過 Hub 運行，承載狀態管理、事件匯流排、規則引擎、場景編排與設備協同等核心模組。
- **Gate** 提供網關接入、Mesh 協同，以及 Runtime island 與設備網路之間的橋接。Gate 使 ART 能接入更廣的協議生態與設備網路，但不取代 Hub 的本地決策中心地位。
- **Sync** 提供 IR / RF 同步接入，將 Runtime 觸達延伸至傳統家電。Sync 使既有設備納入同一 Runtime 語境，擴展 island 的物理覆蓋而不改變 ART 的本地執行邏輯。

這些產品不可互換。Hub 不是 Gate；Gate 不是 Sync。各自承載不同的 Runtime 角色，由 ArtiOS Runtime 統一為連貫的本地執行環境。部署者必須依角色配置節點，而非依產品數量堆疊設備。

ART 的邊界性質體現在三個維度。第一，**執行權威**：本地規則、場景與自動化的最終執行決策，應在 ART 內完成，而非等待 Cloud 回應。第二，**狀態連續性**：設備狀態、事件歷史與 Runtime 日誌在本地維護，確保環境行為可追溯、可恢復、可審計。第三，**降級韌性**：當 Cloud 或 Edge 不可用時，ART 提供離線備援與降級運行，使照明、場景、安防與氣候等關鍵動作仍可執行。這三個維度共同定義空間主權的 operational 含義——不是抽象自治口號，而是可驗證的本地執行能力。

從接口設計看，ART 也是三域之間的「契約執行點」。Cloud 下發的策略、Edge 提供的區域語境、Device 回報的物理狀態，都必須在 ART 內被解析、仲裁與執行。ART 不盲目轉發上游指令，而是在本地規則與安全邊界下，決定哪些指令立即執行、哪些需要本地確認、哪些應在降級模式下拒絕。這種本地仲裁能力是 local-first 架構的核心機制。

這種分離避免系統依賴單一層級。Cloud 可以強化智慧，Edge 可以協同區域，ART 維持本地連續性，設備持續感知與執行。ArtiOS Runtime 邊界是空間主權成為可運作現實之處——房間、家庭或建築區域在此保留行動權威，即使上游協同暫時不可用。理解 ART 邊界，等於理解 OceanAI 如何在分布式架構中保留本地自治：ORT 定義方法，ART 落地執行，Artibird 提供物理載體，三者共同使每個環境都能作為自治的 Runtime island 持續運作。

在實際部署中，ART 邊界也決定了安全與治理的切入點。本地規則的變更、場景的觸發、設備的配對，都應在 ART 內完成身份驗證與權限仲裁，而非完全依賴 Cloud 的即時回應。這使 Runtime island 在網路中斷時仍保有可控性——不是無序的本地自由，而是在 ORT 規則框架下的本地自治。Hub、Gate、Sync 的硬體分工因此不是單純的產品配置，而是 ART 邊界在物理層的可見投影：大腦、橋接、接入，三者缺一不可，各自不可越界。

ART 與 ORT 的關係也應在部署文檔中明確記載：哪些行為遵循 ORT 全局規範，哪些行為允許 island 本地定制。例如，場景觸發的本地延遲要求屬於 ORT 規範，具體場景內容則屬於 island 本地配置。這種全局規範與本地配置的邊界，是 ArtiOS Runtime 作為「邊界層」的第二層含義——它既是 Cloud/Edge 與 Device 之間的執行邊界，也是 ORT 全局規則與 island 本地策略之間的配置邊界。

Runtime 責任邊界的另一實踐意義，在於故障隔離。當 Cloud 服務異常時，ART 應自動進入降級模式，維持本地核心功能；當 Edge 節點失效時，各 island 應斷開跨 island 協同但保持本地運行；當單一 Device 離線時，ART 應更新 island 狀態並觸發預定義的降級策略，而非等待全局超時。責任邊界表因此也是韌性設計的檢查表——每一項職責都應有明確的降級歸屬。ArtiOS Runtime 邊界的設計品質，直接決定了 OceanAI 部署在真實物理環境中的可用性與使用者信任。Hub、Gate、Sync 與 ART 的正確協同，是每一個 Runtime island 從「可連網」到「可運行」的關鍵轉折——連網只是接入條件，本地 Runtime 連續性才是 island 存在的充分條件。部署驗收時，應以 ART 能否在斷網條件下完成核心場景為準，而非僅以設備在線率為準。這是 ArtiOS Runtime 邊界在工程實踐中的具體驗收標準，也是 local-first 承諾是否落地的可驗證指標。

## **3.4 Cross-Domain Runtime Flow｜跨域 Runtime 流程**

**EN**

Understanding domains and boundaries is necessary but not sufficient. Runtime behavior is continuous—it moves across Cloud, Edge, ArtiOS Runtime, and Device as state changes, events propagate, and actions execute.

A typical OceanAI runtime flow follows this path:

```text
Physical Change
    ↓
Device Sensing
    ↓
ArtiOS Runtime Local State / Event Bus
    ↓
Local Rule / Scene / Automation Decision
    ↓
Edge or Cloud Coordination (if needed)
    ↓
ArtiOS Runtime Execution
    ↓
Device Action
    ↓
Runtime Log / Feedback
```

This flow reveals ART as the local execution center. Edge and Cloud can assist, coordinate, learn, and optimize—but local execution should remain available through ArtiOS Runtime whenever possible.

The flow also maps to the five-step runtime cycle:

| Step | Domain Involvement |
| --- | --- |
| Sense | Device Domain |
| Interpret | ArtiOS Runtime, Edge, or Cloud |
| Coordinate | ArtiOS Runtime rules, Edge regional logic, Cloud AI-assisted logic |
| Act | ArtiOS Runtime execution → Device action |
| Learn | Runtime logs feed Cloud learning; Edge refines regional patterns; ART updates local rules |

Not every event requires Cloud or Edge involvement. Many environmental responses—lighting changes, occupancy reactions, scene execution, panel control—should complete entirely within ArtiOS Runtime and the Device Domain. Cross-domain escalation occurs when the response requires broader context: multi-zone coordination, long-term pattern analysis, remote access, or AI-assisted optimization.

This **local-first** flow discipline ensures that physical environments remain responsive regardless of network conditions. Cloud enhances; Edge coordinates regions; ART executes locally. The cross-domain flow is therefore not a pipeline that always reaches the cloud—it is a decision path that escalates only when local authority requires external context.

**ZH-TW**

理解各域與邊界是必要條件，但不足以描述完整運作。Runtime 行為是連續的——它隨狀態變化、事件傳播與動作執行，在 Cloud、Edge、ArtiOS Runtime 與 Device 之間流動。

典型的 OceanAI Runtime 流程如下：

```text
物理變化
    ↓
設備感知
    ↓
ArtiOS Runtime 本地狀態 / 事件匯流排
    ↓
本地規則 / 場景 / 自動化判斷
    ↓
必要時進入 Edge 或 Cloud 協同
    ↓
ArtiOS Runtime 執行
    ↓
設備動作
    ↓
Runtime 日誌 / 回饋
```

這一流程揭示 ART 是本地執行中心。Edge 與 Cloud 可以提供輔助、協同、學習與優化——但本地執行應盡可能透過 ArtiOS Runtime 保持可用。

流程也對應五階段 Runtime 循環：

| 階段 | 域參與 |
| --- | --- |
| 感知（Sense） | Device Domain |
| 理解（Interpret） | ArtiOS Runtime、Edge 或 Cloud |
| 協同（Coordinate） | ART 規則、Edge 區域邏輯、Cloud AI 輔助邏輯 |
| 執行（Act） | ArtiOS Runtime 執行 → 設備動作 |
| 學習（Learn） | Runtime 日誌餵入 Cloud 學習；Edge 精煉區域模式；ART 更新本地規則 |

並非每個事件都需要 Cloud 或 Edge 參與。許多環境回應——照明變化、人在回應、場景執行、面板控制——應完全在 ArtiOS Runtime 與 Device Domain 內完成。跨域升級僅在回應需要更廣語境時發生：多區域協同、長期模式分析、遠端接入或 AI 輔助優化。

**local-first** 流程紀律確保物理環境無論網路條件如何都保持回應能力。Cloud 強化，Edge 協同區域，ART 本地執行。跨域流程因此不是必然抵達雲端的管道——而是僅在本地權威需要外部語境時才升級的決策路徑。

從事件生命週期看，一次典型的環境回應經歷以下階段。感知階段由 Device Domain 完成：感測器偵測變化，設備回報狀態。理解階段由 ART 優先處理：本地規則引擎解析事件語境，判斷是否需要外部協同。協同階段按需啟動：若涉及跨房間調度，Edge 介入；若涉及長期模式或遠端指令，Cloud 介入。執行階段回到 ART：無論上游如何協同，最終動作由 ArtiOS Runtime 在本地觸發。學習階段分散回饋：本地日誌更新 ART 規則，區域模式更新 Edge 策略，全局模式更新 Cloud 模型。

這種設計使 Runtime 流程具備可預測性。開發者與部署者可以明確知道：哪些行為在本地閉環完成，哪些行為需要跨域升級，升級的觸發條件是什麼。對物理環境而言，可預測性等同於可靠性——使用者不應因為雲端延遲而看到照明延後回應，也不應因為 Edge 不可用而失去場景執行能力。

跨域流程還體現了 OceanAI 對「持續運行」的架構承諾。Runtime 不是間歇性的請求-回應，而是狀態與事件的連續流。ArtiOS Runtime 作為本地執行中心，確保這股流在環境內部始終暢通；Cloud 與 Edge 則在需要時注入更廣的智慧，而非取代本地連續性。

以具體場景說明：使用者按下本地面板觸發「離家場景」，事件在 Device Domain 產生，進入 ART 事件匯流排；ART 規則引擎在本地解析場景定義，向相關設備下達指令，全程無需 Cloud 參與。若同一場景需同步關閉另一樓層的設備，ART 向 Edge 請求區域協同；Edge 調度相關 island 的 ART 執行本地動作，仍保持各 island 的本地執行權威。若使用者從外部 App 遠端觸發，Cloud 接收指令後下發至目標 island 的 ART，由 ART 在本地仲裁並執行——Cloud 傳遞意圖，ART 保留執行權。這三種路徑共用同一套跨域流程骨架，差異僅在協同升級的深度。

跨域流程的設計也影響可觀測性。Runtime 日誌在 ART 本地生成，按需同步至 Edge 或 Cloud；這使故障排查可以從本地開始，逐步向上追溯，而非只能從雲端向下猜測。local-first 因此不只是執行策略，也是運維策略——本地日誌、本地狀態、本地規則版本，構成 island 自我描述能力的基礎。

從性能角度看，跨域流程的 local-first 紀律也直接影響使用者體驗。本地閉環的照明、場景與安防回應，延遲應在毫秒至百毫秒級；跨 Edge 的區域協同，延遲可接受在百毫秒至秒級；跨 Cloud 的遠端指令，延遲可能達秒級但不應阻塞本地並行處理。流程設計應預設本地路徑為主路徑，跨域路徑為增強路徑，避免將高頻環境事件預設路由至 Cloud。這是 OceanAI 與傳統雲控智慧家居在流程架構上的根本差異。

跨域流程還定義了「意圖」的傳播語義。使用者或系統產生的意圖，進入 ART 後被解析為本地可執行的動作序列；若需跨域協同，意圖被分解為協同請求而非原始指令的透傳。Cloud 下發的「開啟全屋照明」因此不是六条獨立指令，而是被各 island 的 ART 本地解析為各自場景的協同執行。意圖抽象而非指令透传，是跨域流程保持 local-first 的關鍵機制。跨域 Runtime 流程因此既是行為描述，也是架構約束——它規定了 OceanAI 系統中「誰在何時有權決定、執行與回饋」，使分布式環境中的協同行為可預測、可審計、可降級。流程的正確實現，要求每一個域都嚴守職責邊界：Device 只感知與執行，ART 只本地仲裁與觸發，Edge 只區域協同，Cloud 只全局策略與學習——越界即破壞 local-first 紀律。實務上，跨域流程應被建模為狀態機而非線性管道——每個事件在 ART 內有明確的當前狀態，升級至 Edge 或 Cloud 是狀態轉換而非默認路徑，降級回本地執行也應是正常狀態轉換而非異常恢復。這種狀態機視角，使跨域流程在實現時可被形式化驗證，而非依賴隱式約定。跨域 Runtime 流程的完整描述，因此必須同時包含正常路徑、升級條件與降級策略——三者缺一不可。

## **3.5 Islands Architecture Mapping｜群島模型映射**

**EN**

In the Islands + Swarm Runtime Architecture, each local environment operates as a **Runtime island**—an autonomous execution unit with its own state, rules, and continuity, capable of participating in a larger intelligent network.

ArtiOS Runtime makes this model executable. Without ART, the island remains an architectural metaphor. With ART, each local environment maintains local brain function, network bridge, legacy adapter access, swarm node coordination, runtime memory, and offline autonomy.

| Islands Concept | System Model Mapping |
| --- | --- |
| Island | Room, floor, home, classroom, building zone, campus zone |
| Local Brain | Hub + ArtiOS Runtime |
| Bridge | Gate + ArtiOS Runtime gateway module |
| Legacy Adapter | Sync + ArtiOS Runtime synchronization module |
| Swarm Nodes | Artibird devices and local runtime participants |
| Runtime Memory | ArtiOS Runtime local state, logs, scenes, and event history |
| Offline Autonomy | ArtiOS Runtime local fallback and degraded operation |

Each Runtime island follows **local-first** behavior: critical environmental actions remain available locally whenever possible—lighting control, scene execution, occupancy response, climate coordination, emergency fallback, local panel control, and essential automation. Cloud services improve intelligence, remote access, long-term learning, and multi-site coordination. Edge services improve regional coordination. ART preserves local continuity.

The islands model scales across deployment scenarios. A single home may contain one primary island; a building may contain multiple islands coordinated through Edge; a campus may contain dozens of islands observed and optimized through Cloud. The system model remains consistent—only the coordination scope changes.

ART is the local operating layer that turns distributed intelligent spaces into executable Runtime islands. This mapping connects the abstract islands architecture to the concrete system model of Cloud, Edge, ArtiOS Runtime, and Device domains.

**ZH-TW**

在群島式群集 Runtime 架構中，每個本地環境作為 **Runtime island** 運行——具備自身狀態、規則與連續性的自治執行單元，同時能參與更大的智慧網路。

ArtiOS Runtime 使這一模型具備可執行性。沒有 ART，島嶼仍只是架構隱喻。有了 ART，每個本地環境才能維持本地大腦功能、網路橋接、傳統設備接入、群集節點協同、Runtime 記憶與離線自治。

| 群島概念 | 系統模型映射 |
| --- | --- |
| Island｜島嶼 | 房間、樓層、家庭、教室、建築區域、校園區域 |
| Local Brain｜本地大腦 | Hub + ArtiOS Runtime |
| Bridge｜橋接 | Gate + ArtiOS Runtime 網關模組 |
| Legacy Adapter｜傳統設備接入 | Sync + ArtiOS Runtime 同步接入模組 |
| Swarm Nodes｜群集節點 | Artibird 設備與本地 Runtime 參與者 |
| Runtime Memory｜Runtime 記憶 | ArtiOS Runtime 本地狀態、日誌、場景與事件歷史 |
| Offline Autonomy｜離線自治 | ArtiOS Runtime 本地備援與降級運行 |

每個 Runtime island 遵循 **local-first** 行為：關鍵環境動作應盡可能在本地保持可用——照明控制、場景執行、人在回應、氣候協同、緊急備援、本地面板控制與基礎自動化。Cloud 服務強化智慧、遠端接入、長期學習與多場域協同。Edge 服務強化區域協同。ART 維持本地連續性。

群島模型可跨部署場景擴展。單一家庭可能包含一個主要 island；一棟樓宇可能包含多個由 Edge 協同的 island；一座校園可能包含數十個由 Cloud 觀察與優化的 island。系統模型保持一致——僅協同範圍改變。

從空間主權視角看，Runtime island 是 OceanAI 對「環境自治」的具體實現。每個 island 擁有本地狀態與執行權威，不將核心行為委託給遠端。當 island 之間需要協同時，Edge 提供區域調度，Cloud 提供全局觀察——但協同是增強，而非替代本地自治。

Hub 作為 Local Brain，是 island 的認知與決策中心。Gate 作為 Bridge，連接 island 與外部設備網路及其他 island。Sync 作為 Legacy Adapter，將傳統家電納入 island 的 Runtime 語境。Artibird 設備作為 Swarm Nodes，在 island 內分散感知與執行。ART 維護 Runtime Memory 與 Offline Autonomy，確保 island 在任意網路條件下保持連續運行。

這一映射將抽象群島架構連接至 Cloud、Edge、ArtiOS Runtime 與 Device 三域的具體系統模型。理解映射關係，等於理解 OceanAI 如何在分布式空間中同時實現本地自治與群體協同——每個 island 自治運行，多個 island 透過 Edge 與 Cloud 形成更大的智慧網路。ART 是連接抽象與具體的本地運行層，將分布式智慧空間轉化為可執行的 Runtime island。

群島模型的 Swarm 維度強調分布式參與：Artibird 設備作為 Swarm Nodes，在 island 內分散感知與執行，無需等待中心節點輪詢。狀態與事件在節點間流動，協同邏輯在 ART 內聚合，使 island 具備「群集智慧」的運行特徵——不是單一超級節點，而是多節點協同的 Runtime 整體。Runtime Memory 使 island 保有歷史連續性：場景、規則版本、事件序列在本地持久化，支援回溯、恢復與審計。Offline Autonomy 則是 island 韌性的最終保障：當外部連接全部中斷，island 仍以降級模式維持核心環境功能。

部署規模改變時，群島映射保持語義穩定。住宅場景中，一個 island 通常對應一個家庭空間，Hub 為 Local Brain，Gate 與 Sync 擴展覆蓋。樓宇場景中，每層或每區可能形成獨立 island，Edge 協調跨層動作如消防聯動或能耗管理。校園場景中，教室、行政區、戶外區域各自為 island，Cloud 提供全局觀察與長期優化。無論規模如何，Hub + ART、Gate + ART、Sync + ART 的映射關係不變——變的是 Edge 與 Cloud 的協同半徑。這種語義穩定性，使 OceanAI 系統模型能在不同部署密度下保持一致的設計語言。

群島映射還為新產品接入提供定位框架。任何新的 Artibird 節點接入時，首先應明確它在 island 中的角色：是 Swarm Node 參與分布式感知與執行，還是擴展 Local Brain 的算力，還是強化 Bridge 的協議覆蓋，還是延伸 Legacy Adapter 的接入範圍。角色定位錯誤是部署混亂的常見根源——例如將應作為 Bridge 的 Gate 誤配置為決策中心，或將應作為 Swarm Node 的感測器誤期望其承載協同邏輯。群島映射表因此也是產品接入的決策檢查表。

Offline Autonomy 作為群島映射的最後一項，值得特別強調。它不只是「斷網能用」的口號，而是 Runtime island 在完全失去 Cloud 與 Edge 連接時，仍能以預定義的降級策略維持環境安全與基本舒適的結構性能力。ART 在此維護一份降級策略表：哪些場景仍可用、哪些自動化仍運行、哪些遠端功能暫停、哪些狀態僅本地可見。這份策略表應在部署時明確配置，而非在斷網時臨時 improvised。群島模型映射因此連接架構隱喻與工程實踐——每一個群島概念都有對應的系統組件、硬體角色與 Runtime 職責，使「Runtime island」從比喻成為可部署、可驗證、可運維的架構單元。Swarm 與 island 的結合，也意味著 OceanAI 不追求單一超級節點的集中智慧，而追求多節點協同的分布式 Runtime——這與空間本身的分布式物理特性天然對齊。群島映射的最終價值，在於使架構師與部署者能用同一套語彙描述從單房間到整棟建築的任何 OceanAI 環境，而不必為每種規模重新發明結構。local-first 作為群島行為準則，意味著每個 island 的預設運行模式是自治，參與更大網路是增強選項——這與 Cloud 強化、Edge 協同、ART 執行的三域分工完全一致。Runtime island 不是被動的受控單元，而是主動的 Runtime 主體——這是群島模型區別于傳統分區控制的核心思想。映射表應作為每個新 island 部署的標準檢查項，確保 Local Brain、Bridge、Legacy Adapter 等角色均已正確就位。群島模型映射是 OceanAI 架構從概念到硬體的最後一哩路，不可或缺。

## **4.3 Messaging Framework｜訊息通訊框架**

**EN**  
OceanAI uses a message-driven communication model based on publish–subscribe mechanisms, typically implemented via MQTT.

- Coordination layers subscribe and process relevant information
    
- Commands are distributed through structured topics
    

This framework decouples system components and enables scalable communication across nodes.

**ZH-TW**  
OceanAI 採用基於發布–訂閱（publish–subscribe）的訊息驅動通訊模型，通常透過 MQTT 實現：

- 設備發布狀態與事件
    
- 協同層訂閱並處理相關資訊
    
- 控制指令透過結構化主題分發
