## **10.0 Chapter Purpose｜本章目的**

**EN**  
This chapter defines the Artibird product ecosystem as the physical device system that carries ArtiOS Runtime capabilities inside real environments.

Artibird products are not presented as isolated devices or a simple hardware catalog. They are runtime participants that provide sensing, control, execution, gateway access, synchronization, human interaction, and local runtime hosting.

OceanAI defines the spatial runtime architecture.
ArtiOS Runtime makes OceanAI Runtime locally executable.
Artibird products provide the physical hardware form that allows ArtiOS Runtime capabilities to operate in real spaces.

**ZH-TW**  
本章定義 Artibird 產品生態，作為在真實環境中承載 ArtiOS Runtime 能力的實體設備系統。

Artibird 產品不應被呈現為孤立設備或單純硬體清單。它們是 Runtime 參與者，提供感知、控制、執行、網關接入、同步接入、人機互動與本地 Runtime 承載能力。

OceanAI 定義空間 Runtime 架構。
ArtiOS Runtime 使 OceanAI Runtime 在本地可執行。
Artibird 產品提供實體硬體形態，使 ArtiOS Runtime 能力能夠在真實空間中運行。

## **10.1 Product System Overview｜產品系統總覽**

**EN**  
The Artibird product ecosystem represents the physical realization of OceanAI's system architecture and the hardware carrier of ArtiOS Runtime local runtime execution.

Each Artibird product is designed as a system node with a defined role. These roles include local runtime hosting, network access, legacy synchronization, sensing, user interaction, physical execution, power control, and environmental feedback.

Products are not treated as independent units. They are coordinated through ArtiOS Runtime state, events, rules, scenes, protocol abstraction, offline fallback, and runtime logs.

**ZH-TW**  
Artibird 產品生態是 OceanAI 系統架構的實體化呈現，也是 ArtiOS Runtime 本地 Runtime 執行能力的硬體承載系統。

每個 Artibird 產品都被設計為具備明確角色的系統節點。這些角色包括本地 Runtime 承載、網路接入、傳統設備同步、環境感知、人機互動、物理執行、電力控制與環境回饋。

產品不作為獨立單元存在，而是透過 ArtiOS Runtime 的狀態、事件、規則、場景、協議抽象、離線備援與 Runtime 日誌進行協同。

Sense nodes capture environmental data such as presence, light, temperature, humidity, air quality, and other spatial signals. Vision nodes provide visual context, detection, and AI-assisted environmental understanding.

These devices do not simply report data. They generate runtime events and state inputs that allow ArtiOS Runtime and OceanAI intelligence layers to interpret the environment.

Sense 節點採集人在、光照、溫度、濕度、空氣品質與其他空間訊號。Vision 節點提供視覺情境、偵測與 AI 輔助環境理解。

這些設備不只是回報資料。它們產生 Runtime 事件與狀態輸入，使 ArtiOS Runtime 與 OceanAI 智慧層能夠理解環境。

## **10.9 Panel, Switch, and App as Intent Interfaces｜Panel、Switch 與 App 作為意圖界面**

**EN**  
Artibird Panel, Artibird Switch, and the App provide human interaction with the runtime system.

They do not replace ArtiOS Runtime. They read runtime state from ArtiOS Runtime, send user intent into ArtiOS Runtime, and provide feedback to users.

| Interface | Runtime Role                                                  |
| --------- | ------------------------------------------------------------- |
| Panel     | Local scene control, state display, immediate feedback        |
| Switch    | Physical trigger, direct control, local event generation      |
| App       | Device management, scene management, remote and local control |

User interaction is interpreted as intent. ArtiOS Runtime then determines how the intent should be translated into local runtime behavior based on state, rules, permissions, and system context.

**ZH-TW**  
Artibird Panel、Artibird Switch 與 App 提供人與 Runtime 系統之間的互動。

它們不取代 ArtiOS Runtime。它們讀取 ArtiOS Runtime 的 Runtime 狀態，將使用者意圖送入 ArtiOS Runtime，並向使用者提供回饋。

| 界面          | Runtime 角色        |
| ----------- | ----------------- |
| Panel｜智能面板  | 本地場景控制、狀態顯示、即時回饋  |
| Switch｜智能開關 | 物理觸發、直接控制、本地事件生成  |
| App｜應用程式    | 設備管理、場景管理、遠端與本地控制 |

使用者互動會被理解為意圖。ArtiOS Runtime 再根據狀態、規則、權限與系統情境，判斷該意圖應如何轉化為本地 Runtime 行為。

## **10.10 Plug and Act as Execution Carriers｜Plug 與 Act 作為執行承載體**

**EN**  
Artibird Plug and Artibird Act provide deterministic physical execution.

Plug controls power delivery and may report electrical state, energy usage, or device activity. Act provides relay-based physical actuation for controlled circuits, appliances, or infrastructure nodes.

In the runtime model, these products are execution carriers. They receive actions through ArtiOS Runtime, execute them predictably, report state changes, and participate in runtime logs.

**ZH-TW**  
Artibird Plug 與 Artibird Act 提供確定性的物理執行能力。

Plug 控制電力供應，並可回報電氣狀態、能源使用或設備活動。Act 則透過繼電能力，對受控電路、家電或基礎設施節點進行物理執行。

在 Runtime 模型中，這些產品是執行承載體。它們透過 ArtiOS Runtime 接收動作、以可預期方式執行、回報狀態變化，並參與 Runtime 日誌。

## **10.13 Cross-Device Coordination｜跨設備協同**

**EN**  
Cross-device coordination is mediated through ArtiOS Runtime.

A sensor does not directly control a relay. A panel does not directly redefine a room. A switch does not directly manage a scene. Instead, their inputs enter ArtiOS Runtime as state, events, or intent. ArtiOS Runtime then determines the appropriate local runtime behavior.

For example:

```text
Sense Event
    ↓
ArtiOS Runtime Event Bus
    ↓
Device State Manager
    ↓
Rule / Scene / Automation Engine
    ↓
Execution through Plug / Act / Switch / Sync
    ↓
Runtime Logs
```

This makes coordination traceable, governable, and consistent across device categories.

**ZH-TW**  
跨設備協同透過 ArtiOS Runtime 完成。

感測器不直接控制繼電器。面板不直接重新定義房間。開關不直接管理場景。它們的輸入會作為狀態、事件或意圖進入 ArtiOS Runtime。ArtiOS Runtime 再判斷合適的本地 Runtime 行為。

例如：

```text
感測事件
    ↓
ArtiOS Runtime 事件匯流排
    ↓
設備狀態管理器
    ↓
規則 / 場景 / 自動化引擎
    ↓
透過 Plug / Act / Switch / Sync 執行
    ↓
Runtime 日誌
```

這讓協同行為在不同設備類別之間保持可追蹤、可治理與一致。

**EN**  
This chapter defines how OceanAI is deployed across real environments.

## **11.1 Deployment Model Overview｜部署模型總覽**

**EN**  
The OceanAI deployment model follows a runtime-island structure.

```text
Deployment = one or more Runtime Islands
Runtime Island = Hub + ArtiOS Runtime + Artibird Nodes
Multiple Islands = coordinated through Gate / Edge / Cloud
```

This model allows the same system logic to operate across homes, classrooms, buildings, campuses, and distributed environments.

The scale changes, but the deployment principle remains consistent:

1. Each local environment becomes a runtime island.

2. Each island runs locally through Hub + ArtiOS Runtime.

3. Artibird nodes provide sensing, control, execution, and feedback.

4. Gate connects device networks and runtime islands.

5. Edge and Cloud coordinate larger environments when needed.

**ZH-TW**  
OceanAI 的部署模型採用 Runtime 島嶼結構。

```text
部署 = 一座或多座 Runtime 島嶼

多座島嶼 = 透過 Gate / Edge / Cloud 協同
```

這一模型讓同一套系統邏輯可以運行於住宅、教室、樓宇、校園與分布式環境。

1. 每個本地環境成為一座 Runtime 島嶼。

2. 每座島嶼透過 Hub + ArtiOS Runtime 本地運行。

**EN**  
All communication paths in OceanAI must be protected according to their role.

Communication occurs within islands, across islands, between Gate and Edge, and between Edge and Cloud. Each path has different security requirements.

| Communication Path    | Security Requirement                                             |
| --------------------- | ---------------------------------------------------------------- |
| Device → Hub / ArtiOS Runtime | Authenticated device participation and protected local messaging |
| Panel / App → ArtiOS Runtime  | User-authenticated control and intent submission                 |
| Sync / Gate → Hub     | Secure protocol bridging and trusted event forwarding            |
| Island → Edge         | Protected cross-island coordination                              |
| Edge → Cloud          | Encrypted system coordination and policy synchronization         |
| Cloud → Site          | Authenticated remote management and auditable commands           |

Secure communication ensures confidentiality, integrity, authenticity, and traceability of system actions.

**ZH-TW**  
OceanAI 中的所有通訊路徑都必須根據其角色受到保護。

通訊發生於島嶼內部、跨島嶼之間、Gate 與 Edge 之間，以及 Edge 與 Cloud 之間。每條路徑的安全要求不同。

| 通訊路徑                 | 安全要求             |
| -------------------- | ---------------- |
| 設備 → Hub / ArtiOS Runtime    | 已認證設備參與與受保護的本地訊息 |
| Panel / App → ArtiOS Runtime | 使用者認證控制與意圖送入     |
| Sync / Gate → Hub    | 安全協議橋接與可信事件轉發    |
| 島嶼 → Edge            | 受保護的跨島協同         |
| Edge → Cloud         | 加密系統協同與策略同步      |
| Cloud → 場域           | 已認證遠端管理與可審計指令    |

安全通訊確保系統行為的保密性、完整性、真實性與可追蹤性。

## **12.9 Cross-Island Auditability｜跨島嶼可審計性**

**EN**  
Cross-island behavior must be auditable.

When a command, policy, scene, automation, or emergency behavior affects multiple runtime islands, the system must record what happened, where it happened, why it happened, which entities participated, and whether the result matched expectations.

A cross-island audit record may include:

| Audit Field       | Description                                                  |
| ----------------- | ------------------------------------------------------------ |
| Timestamp         | When the behavior occurred                                   |
| Site ID           | Home, building, campus, or organization                      |
| Island IDs        | Runtime islands affected                                     |
| Trigger           | Schedule, policy, user action, sensor event, emergency event |
| Decision source   | Rule, scene, cloud policy, edge coordination, user command   |
| Participants      | Hub, Gate, Edge, Cloud, Artibird nodes                       |
| Permission result | Whether the action was authorized                            |
| Execution result  | Success, partial success, failure, timeout                   |
| Override record   | Whether a user or operator intervened                        |
| Sync status       | Whether records were synchronized to cloud                   |

Auditability turns system behavior into accountable operation.

**ZH-TW**  
跨島嶼行為必須可審計。

當一個指令、策略、場景、自動化或緊急行為影響多座 Runtime 島嶼時，系統必須記錄發生了什麼、發生在哪裡、為什麼發生、哪些實體參與，以及結果是否符合預期。

跨島審計記錄可包含：

| 審計欄位                   | 說明                              |
| ---------------------- | ------------------------------- |
| Timestamp｜時間戳          | 行為發生時間                          |
| Site ID｜場域 ID          | 家庭、樓宇、校園或組織                     |
| Island IDs｜島嶼 ID       | 受影響的 Runtime 島嶼                 |
| Trigger｜觸發來源           | 日程、策略、使用者動作、感測事件、緊急事件           |
| Decision source｜決策來源   | 規則、場景、雲端策略、邊緣協同、使用者指令           |
| Participants｜參與者       | Hub、Gate、Edge、Cloud、Artibird 節點 |
| Permission result｜權限結果 | 動作是否已授權                         |
| Execution result｜執行結果  | 成功、部分成功、失敗、逾時                   |
| Override record｜覆蓋記錄   | 是否有使用者或運維者介入                    |
| Sync status｜同步狀態       | 記錄是否已同步至雲端                      |

可審計性使系統行為成為可追責的運行過程。

## **12.10 Cloud-Level Governance｜雲端級治理**

**EN**  
Cloud-level governance defines system-wide policies, roles, visibility, records, and remote administration.

Cloud governance should not eliminate local autonomy. Instead, it provides policy structure, long-term accountability, multi-site visibility, and administrative control.

Cloud governance includes:

| Governance Capability   | Role                                                                       |
| ----------------------- | -------------------------------------------------------------------------- |
| Policy management       | Defines system-wide rules, constraints, and allowed behaviors              |
| Role management         | Defines user, administrator, installer, teacher, operator, and guest roles |
| Site management         | Manages homes, buildings, campuses, and organizations                      |
| Audit repository        | Stores synchronized local and cross-island audit records                   |
| Remote administration   | Allows authorized remote configuration and support                         |
| Analytics and reporting | Supports operational insight and long-term optimization                    |
| Compliance support      | Supports governance requirements for schools, buildings, or organizations  |

Cloud governs the system.
ArtiOS Runtime preserves local runtime continuity.
Edge coordinates regional behavior.
Devices execute physical actions.

**ZH-TW**  
雲端級治理定義系統級策略、角色、可視化、記錄與遠端運維。

雲端治理不應消除本地自治，而是提供策略結構、長期可追責性、多場域可視化與管理控制。

雲端治理包括：

| 治理能力  | 角色                         |
| ----- | -------------------------- |

| 場域管理  | 管理家庭、樓宇、校園與組織              |
| 審計資料庫 | 儲存同步後的本地與跨島審計記錄            |
| 遠端運維  | 允許授權遠端配置與支援                |
| 分析與報告 | 支援運行洞察與長期優化                |
| 合規支援  | 支援學校、樓宇或組織的治理要求            |

Cloud 治理系統。
ArtiOS Runtime 維持本地 Runtime 連續性。
Edge 協同區域行為。
設備執行物理動作。

## **12.11 Firmware and Update Governance｜韌體與更新治理**

**EN**  
Firmware and software updates must be governed because they affect local runtime behavior.

Updates may apply to Hub, Gate, Sync, panels, sensors, execution devices, ArtiOS Runtime modules, Edge services, or cloud services. Each update should be authenticated, validated, staged, monitored, and reversible when possible.

| Update Requirement  | Description                                               |
| ------------------- | --------------------------------------------------------- |
| Signed updates      | Update packages must be signed and verified               |
| Source validation   | Devices and services must validate update sources         |
| Compatibility check | Updates should be checked against island configuration    |
| Staged rollout      | Updates may be applied gradually across islands           |
| Rollback support    | Critical components should support rollback when possible |
| Runtime safety      | Updates should not break essential local behavior         |
| Audit record        | Update history should be recorded and traceable           |

**ZH-TW**  
韌體與軟體更新必須受到治理，因為它們會影響本地 Runtime 行為。

更新可能作用於 Hub、Gate、Sync、面板、感測器、執行設備、ArtiOS Runtime 模組、Edge 服務或雲端服務。每次更新都應經過認證、驗證、分階段部署、監測，並在可能時支援回滾。

| 更新要求       | 說明               |
| ---------- | ---------------- |
| 簽名更新       | 更新包必須經簽名與驗證      |
| 來源驗證       | 設備與服務必須驗證更新來源    |
| 相容性檢查      | 更新應根據島嶼配置進行相容性檢查 |
| 分階段部署      | 更新可在多座島嶼中逐步部署    |
| 回滾支援       | 關鍵組件應盡可能支援回滾     |
| Runtime 安全 | 更新不應破壞關鍵本地行為     |
| 審計記錄       | 更新歷史應被記錄並可追蹤     |

## **12.12 Fail-Safe and Emergency Behavior｜故障安全與緊急行為**

**EN**  
Runtime islands must support fail-safe behavior.

A failure may occur in devices, Hub, Gate, local network, Edge, Cloud, power, firmware, or coordination logic. The system should define what remains available, what is restricted, and what enters safe mode.

| Failure Condition | Expected Behavior                                                             |
| ----------------- | ----------------------------------------------------------------------------- |
| Cloud unavailable | Local ArtiOS Runtime behavior continues where possible                                |
| Edge unavailable  | Islands continue local execution and sync later                               |
| Gate unavailable  | Local island behavior continues if Hub and devices remain reachable           |
| Hub degraded      | Essential device-level safety behavior should remain available where possible |
| Device offline    | State is marked, scenes or rules adapt or fail safely                         |
| Network unstable  | Commands are retried, delayed, or locally constrained                         |
| Emergency event   | Safety policy overrides normal automation                                     |

Fail-safe design ensures that intelligent environments remain safe under partial failure.

**ZH-TW**  
Runtime 島嶼必須支援故障安全行為。

故障可能發生於設備、Hub、Gate、本地網路、Edge、Cloud、電力、韌體或協同邏輯。系統應定義哪些能力保持可用，哪些能力受到限制，哪些行為進入安全模式。

| 故障條件      | 預期行為                 |
| --------- | -------------------- |
| Cloud 不可用 | 本地 ArtiOS Runtime 行為盡可能持續    |
| Edge 不可用  | 島嶼保持本地執行，稍後同步        |
| Gate 不可用  | 若 Hub 與設備仍可達，島嶼內行為持續 |
| Hub 降級    | 關鍵設備級安全行為盡可能保持可用     |
| 設備離線      | 標記狀態，場景或規則調整或安全失敗    |
| 網路不穩      | 指令重試、延遲或受本地限制        |
| 緊急事件      | 安全策略覆蓋一般自動化          |

故障安全設計確保智慧環境在部分故障下仍保持安全。
